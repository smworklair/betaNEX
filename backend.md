# backend.md — что нужно реализовать на бэкенде для расширения betaNEX

Этот документ описывает бэкенд-работу под расширение функциональности,
которое **уже собрано во фронтенде как бета** (флаг `beta`, данные живут в
`localStorage` через `web/src/beta/store.ts`). Задача бэкенда — заменить
локальный слой на реальные модули `nexd`, сохранив те же формы данных и
операции, чтобы UI-компоненты не переписывались (как уже сделано для
`tasks` и `campus`).

## Как это устроено сейчас (чтобы не сломать контракт фронта)

- **Локальный слой беты.** `web/src/beta/store.ts` даёт `useCollection<T>(key, seed)`
  с CRUD + массовыми операциями поверх `localStorage` (ключи `nex-beta:*`).
  Универсальные экраны (`web/src/beta/manager.tsx`) рисуют таблицу + поиск +
  создание/редактирование/удаление + экспорт CSV + импорт JSON.
- **Точки замены.** Чтения оборачиваются в `withFallback()` (`web/src/api/client.ts`):
  сперва реальный вызов, при ошибке — мок. Мутации идут напрямую и должны
  отражать ответ сервера. Новые модули должны повторить этот паттерн: добавить
  `web/src/api/<module>.ts`, а компоненты переключить со `useCollection` на
  api-обёртку.
- **Общие правила.** Всё под `/api/v1`, аутентификация обязательна,
  мультиарендность (`tenant_id` в каждой таблице и во всех запросах), ошибки —
  RFC 9457 (`application/problem+json`, `httpapi.WriteProblem`), идемпотентность
  мутаций через `Idempotency-Key` (миграция `00006_idempotency.sql` уже есть),
  все доменные действия пишутся в журнал аудита как отдельный участник.

## Общая архитектура для новых модулей

Каждый раздел — отдельный модуль `internal/module/<name>` по образцу
`internal/module/tasks`:

```
internal/module/<name>/
  <name>.go     // доменные типы, команды (Create/Update/Delete/…), инварианты
  repo.go       // Repository поверх sqlc (pgx), фильтры, пагинация, tenant-scope
  http.go       // Routes(bus, repo) → REST под /api/v1/<name>
  <name>_test.go
migrations/000NN_<name>.sql   // таблицы + индексы (goose/sql), FK на tenant
```

Обязательные сквозные требования для **всех** модулей ниже:

- **CRUD**: `POST` (create), `GET` (list с `?q=&sort=&limit=&offset=` и фильтрами),
  `GET /{id}`, `PATCH /{id}` (partial update), `DELETE /{id}`.
- **Массовые операции**: `POST /api/v1/<name>/bulk` с телом
  `{ ids: [...], op: "delete" | "patch" | "complete" | ..., patch?: {...} }`.
- **Экспорт/импорт**: `GET /api/v1/<name>/export?format=csv|xlsx` и
  `POST /api/v1/<name>/import` (CSV/JSON, валидация, отчёт об ошибках построчно).
- **История изменений**: таблица `<name>_history` (кто, что, когда, дифф) или
  общий `audit_log`; отдаётся `GET /api/v1/<name>/{id}/history`.
- **Права**: проверка через `internal/kernel/authz` по ролям и разрешениям
  (см. раздел 7). Матрица «роль × разрешение» — источник истины доступа.
- **Поиск и фильтрация**: полнотекстовый индекс (см. `00003_search.sql`) +
  предметные фильтры (статус, дата, категория, ответственный).

Ниже — по разделам: сущности, эндпоинты, поля, фоновые задачи и интеграции.

## 1. Задачи (Tasks) — расширение существующего модуля

Модуль `tasks` уже есть, но урезан (title/note/due/assignee/status). Нужно
расширить домен и схему до модели фронта (`web/src/pages/tasks.tsx`).

**Расширить сущность `task`:** `priority (low|normal|high|urgent)`,
`status (open|in_progress|done|canceled)`, `category`, `tags[]`, `due_on`,
`assignees[]`, `watchers[]`, `recurrence (none|daily|weekly|monthly)`.

**Дочерние сущности (отдельные таблицы, FK на task):**
- `task_subtasks` (title, done, order)
- `task_checklist` (text, done, order)
- `task_comments` (author_id, text, created_at)
- `task_attachments` (использовать модуль `files`, `00004_files.sql`)
- `task_history` (событие, дифф, actor, at) — заполняется на каждое изменение

**Шаблоны и правила:**
- `task_templates` (name, title, note, priority, category, checklist[]).
  `POST /api/v1/tasks/templates`, `POST /api/v1/tasks/from-template/{id}`.
- `task_rules` (name, trigger, action, assignee, priority, enabled) —
  **автосоздание задач по событиям**. Правила исполняет фоновый воркер,
  подписанный на доменные события (просрочка платежа, посещаемость < N%,
  новый приказ). Реализовать через шину событий + планировщик (cron для
  временных правил, подписка на события для реактивных).

**Рассылка задач (send):** `POST /api/v1/tasks/{id}/dispatch`
с телом `{ mode: "one"|"group"|"many", user_id?, group_id?, user_ids?[] }` —
создаёт назначения/копии и уведомления получателям (email + внутренние
уведомления). Массовая рассылка группе разворачивается в назначения по
участникам группы.

**Повторяющиеся задачи:** воркер по `recurrence` создаёт следующий экземпляр
при закрытии/по расписанию.

**Эндпоинты:** к текущим добавить `PATCH /{id}`, `POST /{id}/dispatch`,
подресурсы `subtasks|checklist|comments|attachments|history`, `templates`,
`rules`, `POST /bulk`.

## 2. Календарь (Calendars)

Новый модуль `calendar`. Фронт: `web/src/pages/calendar.tsx`.

**Сущности:**
- `calendar_events` (title, kind: meet|exam|deadline|lesson|personal, starts_at,
  ends_at, location/room_id, recurrence, reminder, note, created_by).
- `event_participants` (event_id, user_id, response: pending|accepted|declined).
- `event_groups` (event_id, group_id) — привязка учебных групп для массовых приглашений.
- `event_attachments` (модуль `files`).
- `rooms` (аудитории) + `room_bookings` (event_id, room_id, interval) с проверкой
  **пересечений броней** (exclusion constraint по времени, `tstzrange`).

**Функции бэкенда:**
- **Приглашения и массовая рассылка:** `POST /api/v1/calendar/events/{id}/invite`
  `{ user_ids?[], group_ids?[] }` → уведомления + записи участников. Разворачивание
  групп в участников на стороне сервера.
- **Повторяющиеся события:** генерация экземпляров (RRULE-подобно) или разворачивание
  на лету при выборке диапазона.
- **Напоминания:** воркер шлёт уведомления `reminder` (10м/1ч/1д) до `starts_at`.
- **Бронирование аудиторий:** атомарная проверка занятости, ответ 409 при конфликте.
- **Связь с задачами:** флаг `link_task` → создаёт связанную задачу-напоминание
  (кросс-модульная команда в `tasks`).

**Эндпоинты:** CRUD событий, `GET /events?from=&to=&kind=`, `/invite`, `/rooms`,
`/rooms/{id}/availability`, `POST /events/bulk`.

## 3. Финансы (Finance) — уровень «как в 1С»

Модуль `finance` есть (счета/платежи по `00002_finance.sql`). Расширить до
полноценного учёта. Фронт: `web/src/pages/finance-beta.tsx`.

**Справочники:** `counterparties` (контрагенты: студенты, ЮЛ, поставщики, ИНН/КПП),
`chart_of_accounts` (план счетов), `articles` (статьи ДДС), `bank_accounts`, `cash_registers`.

**Документы (каждый — таблица + номер + статус + проведение):**
- `invoices` — счета на оплату (number, counterparty, sum, date, status).
- `vat_invoices` — счета-фактуры (sum, vat, книга покупок/продаж).
- `acts` — акты выполненных работ (subject, sum, статус подписи).
- `contracts` — договоры (subject, sum, срок, статус исполнения; связь со студентами/поставщиками).
- `payments` — платежи (расширить существующие).
- `cash_ops` — касса: ПКО/РКО (приход/расход, статья) → кассовая книга.
- `bank_ops` — банк: платёжные поручения, направление, проведение.

**Учёт и отчётность:**
- `journal_entries` — **журнал операций (проводки, двойная запись)**: дебет/кредит/сумма.
  Документы формируют проводки при проведении (`POST /{doc}/{id}/post`).
- **Дебиторская/кредиторская задолженность** — представления/агрегаты по контрагентам
  и договорам с возрастной структурой (aging) и сроками.
- **Бюджетирование** — `budgets` (план/факт по статьям, периоды), уже частично есть.
- **Отчёты:** ОСВ, реестр платежей, задолженность, начисление стипендий, налоговые
  формы (УСН). `GET /api/v1/finance/reports/{code}?period=` → данные + выгрузка.

**Кросс-функции:**
- **Массовые операции:** массовое проведение/отмена, массовые напоминания должникам.
- **Экспорт/импорт:** выгрузка реестров (CSV/XLSX), импорт банковской выписки
  (1С-совместимый `1CClientBankExchange`, MT940), выгрузка для 1С/ФНС.
- **Журнал операций и поиск** — общий фильтр по дате/контрагенту/сумме/статусу.
- **Права:** разделить «просмотр финансов» и «изменение финансов» (уже в матрице ролей).

**Инварианты:** запрет удаления проведённых документов (только сторно),
контроль баланса проводок, идемпотентность при импорте.

## 4. Учебный процесс (Academics)

Расширить `campus`/добавить `academics`. Фронт: `web/src/pages/academic-beta.tsx`
(+ существующие journal/schedule/attendance/exams).

**Сущности с полным CRUD:**
- `disciplines` (дисциплины: name, code, hours, teacher, control_form).
- `lessons` / `schedule` (занятия и расписание; уже есть каркас) — с проверкой
  конфликтов преподавателей/аудиторий.
- `exams` (сессия: группа, дисциплина, дата, аудитория, допуски).
- `homework` (домашние задания: дисциплина, группа, задание, срок, статус).
- `grades` (оценки; журнал уже частично есть) — история выставления.
- `gradesheets` (ведомости: экзаменационные/зачётные, номер, статус, состав).
- `curricula` (учебные планы: специальность, год, семестры, статус).
- `orders` (приказы по студсоставу: зачисление/отчисление/перевод/академ; статус,
  подпись, связь со студентом; генерация печатной формы).

**Функции:** формирование ведомостей из журнала, расчёт допусков к сессии
(закрытые задолженности), генерация приказов из шаблонов, печать (PDF),
связь оценок ↔ ведомостей ↔ приказов, экспорт/импорт.

## 5. Люди (People)

Расширить `campus`. Фронт: `web/src/pages/people-beta.tsx` + существующие
students/groups/staff. Для **всех** подразделов — create/update/delete.

**Сущности:**
- `students`, `teachers`, `employees` (единый профиль `persons` + роли-привязки).
- `groups`, `departments` (отделения/кафедры/службы, руководитель, тип).
- `curators` (закрепление преподавателя за группой, период).

**Функции:** CRUD с валидацией (уникальность email/номера), связи person↔group↔dept,
история статусов (обучается/академ/отчислен; активен/отпуск/уволен),
массовый импорт (зачисление списком), экспорт, поиск/фильтрация.

## 6. Аналитика (Analytics)

Новый модуль `analytics` (в основном на чтение/агрегатах). Фронт:
`web/src/pages/analytics-beta.tsx`.

**Нужно:**
- **KPI** — агрегаты (средний балл, посещаемость, собираемость оплат, риск отсева),
  с дельтами к прошлому периоду. `GET /api/v1/analytics/kpi?period=`.
- **Графики/разрезы** — данные для столбцов/пончиков/линий по группам/специальностям/периодам.
- **Тепловые карты** — матрица посещаемости группы × недели.
- **Прогнозирование** — сервер считает тренд/прогноз (сейчас на фронте линейная
  экстраполяция; перенести в надёжный расчёт, при необходимости — модель).
- **Конструктор отчётов** — `report_definitions` (метрика, разрез, тип графика,
  фильтры) → `POST /api/v1/analytics/report` возвращает данные по определению.
- **Сохранённые представления** — `saved_views` (per-user), CRUD.
- **Экспорт** — дашборд/отчёт в PDF/XLSX; выгрузка данных.
- **Дашборды** — наборы виджетов (`dashboards`, `dashboard_widgets`).

Реализация: materialized views / предагрегаты для тяжёлых метрик, кэш
(`internal/platform/cache`), пересчёт по расписанию.

## 7. Безопасность (Security)

Новый модуль `security` + расширение `kernel/authz`. Фронт:
`web/src/pages/security-beta.tsx`. ИИ здесь — **один из инструментов**
(детект аномалий), а не основа раздела.

**Пользователи и доступ:**
- `users` (CRUD, статус, привязка к person), `roles`, `permissions`,
  `role_permissions` (**матрица прав** — источник истины авторизации).
  Проверки во всех модулях идут через `authz` по этой матрице.

**Журналы:**
- `audit_log` (уже частично есть, `httpapi.AuditRoutes`) — расширить фильтрами
  (actor/severity/target/date), экспорт.
- `security_events` — журнал безопасности (входы, смены прав, блокировки).

**Сессии и устройства:**
- `sessions` (активные входы, устройство, IP, гео) + `POST /{id}/revoke`.
- `devices` (доверенные/недоверенные), привязка к пользователю.

**Внешний доступ:**
- `api_keys` (выпуск/область/ротация/отзыв; хранить хэш, показывать префикс).
- OAuth-провайдеры (Google/Яндекс) — конфиг и коллбэки.
- SSO (SAML 2.0 / OIDC) — интеграция с корпоративным IdP.
- MFA (TOTP/SMS) — включение, резервные коды, enforcement по политике.

**Политики и сеть:**
- `security_policies` (обязательная MFA, сложность паролей, таймаут сессии).
- `ip_rules` — **белые/чёрные списки IP/подсетей**, применяются в middleware входа.
- Ограничение доступа по IP (enforcement на уровне шлюза).

**Инфраструктура:**
- `backups` — список копий, ручной запуск (`POST /api/v1/security/backups`),
  восстановление, расписание; интеграция с реальным механизмом бэкапа БД.
- `certificates` — учёт TLS-сертификатов, срок, оповещения об истечении.
- **Мониторинг/целостность** — статусы подсистем (`services`), проверки,
  журналы событий; отдавать в `GET /api/v1/security/health`.

**ИИ-инструмент:** детект аномалий входов/операций как фоновый анализатор,
пишущий наблюдения в `security_events`; не является ядром раздела.

## 8. Интерфейс (Interface) — что нужно от бэкенда

Основное сделано на фронте (прозрачность, горячие клавиши Windows,
полноэкранный режим, скрытие/плавающая/закреплённая боковая панель) и хранится
в `localStorage` (`nex-prefs`). От бэкенда нужно:

- **Синхронизация пользовательских настроек интерфейса между устройствами:**
  `GET/PUT /api/v1/me/preferences` (JSON: accent, density, font, corners, strip,
  solid, sidebar mode, zen, dock, topbar). Сейчас — только локально.

## 9. Работа с текстом (Selection actions) — что нужно от бэкенда

Контекстное меню по выделению (`web/src/ai.tsx`, `SelectionPopover`) уже создаёт
задачи/заметки/события локально и вызывает ИИ. Для полноты нужно:

- **Заметки:** модуль `notes` (`notes` таблица: text, source, created_by) —
  сейчас пишутся в `localStorage` (`nex-beta:notes`).
- **Отправить сообщение / письмо:** проброс выделенного текста в модули
  сообщений и почты (создание черновика с预заполненным телом).
- **Создать задачу / событие из выделения:** уже используют `tasks`/`calendar`
  (см. разделы 1–2) — после появления API писать напрямую туда.
- **Перевод/суммаризация/передача ИИ:** через существующий LLM-слой; при желании
  вынести на сервер (`POST /api/v1/ai/translate|summarize`) для единых лимитов и аудита.

## Порядок внедрения (рекомендация)

1. **Kernel:** роли/разрешения/матрица (раздел 7) — база доступа для остального.
2. **Tasks++** и **Calendar** — самые востребованные, каркас `tasks` уже есть.
3. **Finance++** — документы и проводки (крупнейший объём учёта).
4. **Academics / People** — CRUD-сущности поверх `campus`.
5. **Analytics** — предагрегаты и отчёты поверх готовых данных.
6. **Security (полный)**, **preferences**, **notes/ai-эндпоинты**.

Для каждого шага: сначала правим `api/openapi.yaml` (источник истины),
затем генерируем клиент/сервер, затем добавляем `web/src/api/<module>.ts`
и переключаем экраны со `useCollection` на api-обёртку через `withFallback`.
