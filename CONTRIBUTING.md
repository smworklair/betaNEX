# Как работать с репозиторием NEX

## Быстрый старт

```sh
cp .env.example .env
make dev        # поднять Postgres (docker compose)
make run        # запустить nexd из исходников
make test       # тесты с race-детектором
make lint       # golangci-lint
```

Hot-reload: `make watch` (требуется установленный air).

## Правила

- **Изменения данных — только через шину команд** (internal/kernel/command). Прямые UPDATE из хендлеров запрещены.
- **Сначала контракт:** изменения API начинаются с `api/openapi.yaml`, затем кодогенерация, затем реализация.
- **Миграции не редактируются** после попадания в main — только новые поверх (см. migrations/README.md).
- **Каждая доменная таблица** несёт `tenant_id` и RLS-политику.
- Зависимости направлены внутрь: приложения → модули → ядро. Модули не зависят друг от друга.
- PR принимается только с зелёным CI (lint + test) и тестами на новое поведение.
- Комментарии в новых пакетах пишем на русском; существующий английский код правим без смены языка файла.

## Куда что класть

| Что | Куда |
|---|---|
| Доменная логика | `internal/module/<имя>` |
| Общая инфраструктура | `internal/platform/<адаптер>` |
| SQL-запросы | `internal/platform/postgres/queries` (+ `make sqlc`) |
| Миграции | `migrations/` |
| Контракт API | `api/openapi.yaml` |
| Dev-скрипты (Python: смоук API, демо-данные) | `tools/` |
| Документация решений | `docs/decision-log.md` (новый ADR на каждое существенное решение) |
