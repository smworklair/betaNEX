## Слоган (tagline)

**NEX** — модульная платформа для настоящих организационных систем, где ИИ — полноценный актор, а не прикрученный чат.

---

## 1. Что такое NEX?

NEX — web-first платформа для создания специализированных информационных систем на одном переиспользуемом бэкенде. Первый продукт на ней — консоль операций колледжа/вуза, но ядро намеренно **доменно-независимое**: ту же основу можно позже применить к другим организациям (муниципальные сервисы, клиники, логистика и т.д.).

Архитектурно NEX — **модульный монолит**: тонкое, доменно-свободное **ядро** (identity, авторизация, мультитенантность и позвоночник Commands → Events → Audit) хостит независимые **модули** с настоящей бизнес-логикой — финансы, campus, задачи, файлы, уведомления, терминал и др. Конкретные системы — это приложения, которые композируют модули. Зависимости смотрят только внутрь: приложения → модули → kernel; kernel ни от чего «выше» не зависит.

В отличие от многих «AI-продуктов», где LLM приклеен к форме и выдаётся за инновацию, NEX проектирует ИИ как **полноценного системного актора**. Ассистент обязан ходить теми же аутентифицированными, авторизованными, мультитенантными и аудируемыми путями, что и человек. Это не маркетинговая фраза — так устроен поток запросов: React UI → `nexd` (Go) → AI-gateway → бюджеты и провайдеры.

---

## 2. Почему NEX?

Большинство софта для учебных заведений и организаций среднего размера — это либо:

- куча разрозненных инструментов (таблицы, чат-боты, наследие 1С-подобных модулей, почта), либо  
- тяжёлый enterprise-комбайн: дорого, жёстко и враждебно к изменениям.

Параллельно AI-волна породила море демо, которые «отвечают на вопросы», но **ничего безопасно не делают** внутри реальных процессов: нет тенантности, ролей, аудита, контроля бюджета, единого источника правды.

Мы сделали NEX, потому что нужны обе стороны сразу:

1. серьёзный backend-каркас для мультитенантных операционных систем, и  
2. ИИ, который живёт *внутри* этого каркаса — а не рядом с ним.

Первый вертикаль — операции колледжа/вуза: студенты, группы, журнал, финансы, задачи, уведомления, home в духе command center. Цель платформы шире: переиспользовать то же ядро для следующего домена, не переписывая identity, authz, audit и tenancy.

---

## 3. Проблема

Колледжи и похожие организации живут на фрагментированных данных и «знаниях в головах»:

| Боль | Как это выглядит на практике |
| --- | --- |
| Разрозненные системы | Оценки в одном месте, оплаты в другом, задачи в чате, файлы в папках |
| Нет общего контекста | Куратор не видит посещаемость + оценки + долг по одному студенту в один клик |
| Слабая подотчётность | Кто, что и под какой ролью изменил — часто непонятно |
| ИИ как игрушка | Чат-боты без изоляции тенантов, лимитов трат и проверки прав |
| Налог на пересборку | Каждая новая «информационная система» заново пишет логин, роли и логи |

NEX бьёт в структурную проблему: одно платформенное ядро, компонуемые доменные модули и ИИ по тем же правилам, что и все остальные.

## 4. Что умеет система

### Для колледжа (первое приложение)

- **Campus** — группы, студенты, контекст учебного журнала  
- **Финансы** — операции в духе двойной записи и задел под отчётность  
- **Задачи** — назначаемая работа с уведомлениями  
- **Файлы** — метаданные вложений и content-addressable blob-хранилище  
- **Уведомления** — лента системных и доменных событий  
- **Home / command center** — сводка дня, ярлыки, сигналы риска, object drawer  
- **Терминал / админ-консоль** — операционный контур для администраторов  
- **AI-assist** — контекстный мини-чат: понимает *где* пользователь и *какие факты* на экране  

### Для платформы (что переиспользуется)

- Мультитенантная изоляция (tenant-контекст + PostgreSQL RLS)  
- Сессионная auth (argon2id, httpOnly-сессии, rate-limit логина, аудит)  
- RBAC на шине команд (не только в HTTP-хендлерах)  
- Append-only аудит значимых изменений  
- Контракт OpenAPI, ошибки problem+json, идемпотентность  
- Наблюдаемость: structured logs, request_id, Prometheus-метрики  
- AI-gateway с маршрутизацией провайдеров, бюджетами тенанта, лимитами и изоляцией секретов

## 5. Как мы это собрали

NEX — небольшой monorepo с чёткими границами процессов:

| Слой | Технологии | Роль |
| --- | --- | --- |
| **Backend (`nexd`)** | Go, `net/http`, pgx, sqlc, goose | Модульный монолит: kernel + modules + HTTP API |
| **БД** | PostgreSQL 16/17 | Источник истины, RLS, миграции вшиты в бинарник |
| **Frontend (`web/`)** | React, TypeScript, Vite, Tailwind | UI продукта / прототип консоли колледжа, эволюция в полноценный клиент |
| **AI gateway** | Python, FastAPI | Адаптер LLM: бюджеты, лимиты, context registry, мультипровайдер |
| **Infra** | Docker Compose, Caddy, GitHub Actions → GHCR | Локальный стек, prod-shaped деплой, релизы образов по тегу |

### Правила, которых мы реально придерживаемся

1. **Один путь записи** — мутации домена идут через command bus с authz + audit.  
2. **Зависимости внутрь** — модули не импортируют друг друга; kernel не тянет «верхние» слои.  
3. **В браузере нет ключей провайдеров** — UI ходит только в `nexd`; `nexd` аутентифицирует и проксирует AI; секреты живут в gateway.  
4. **Тенант выводится на сервере** — ему нельзя доверять из тела запроса или произвольных заголовков клиента.  
5. **Скучные технологии там, где важна надёжность** — SQL, явные модули, OpenAPI, Prometheus — не зоопарк фреймворков.

Рядом с продакшен-кодом лежат учебные материалы (`docs/learn/`, `beta/FrankAI`, `beta/Nex-pilot`): проект одновременно система и обучаемый стек.

## 6. Архитектура (картинка словами)

```
Браузер (React SPA)
    │  cookie-сессия, same-origin API (или CORS)
    ▼
nexd (Go, модульный монолит)
    ├── kernel: identity · authz · tenancy · command/event/audit
    ├── modules: finance · campus · tasks · files · notifications · terminal
    ├── platform: httpapi · postgres/RLS · blob · cache · outbox · metrics
    └── AI proxy: /api/v1/ai/*  →  auth + trusted tenant
            ▼
      ai-gateway (Python)
            ├── rate limit · budget тенанта · system/context prompt
            └── провайдеры: Gemini / OpenAI-compatible / GigaChat / YandexGPT / …
```

ИИ — не отдельная чёрная дверь сбоку. Это ещё одна capability за той же входной дверью.

---

## 7. Почему важно «ИИ как актор»

Если ассистент читает данные студентов или запускает побочные эффекты, на него должны распространяться:

- **аутентификация** — кто спрашивает  
- **авторизация** — что роли можно видеть и делать  
- **тенантность** — какая граница организации  
- **аудит** — что запросили и что изменилось  
- **бюджет / rate limits** — сколько трат и нагрузки допустимо  

Текущий путь NEX закрепляет это на границе: фронтенд не ходит к провайдерам и gateway напрямую; `nexd` подставляет доверенный tenant из сессии; gateway проверяет shared secret и применяет budget/rate. Дальний roadmap — действия ИИ через ту же command/event-шину, чтобы agent-поведение оставалось проверяемым и отзываемым.

Параллельно есть учебный трек (`FrankAI` + `Nex-pilot`) — маленький путь «своя модель» — чтобы «ИИ внутри NEX» был не только API-клеем, но и инженерной историей, которую можно изучать.

## 8. С какими сложностями столкнулись

- **Прототип vs архитектура** — визуальный UI «КИС Колледж» появился раньше зрелого Go-kernel; пришлось держать его как design reference, не давая диктовать бэкенд.  
- **Безопасная интеграция ИИ** — ранние пути с client-supplied tenant / ключами в браузере были найдены и перепроектированы.  
- **Дисциплина модульного монолита** — удержаться и от «всё в хендлерах», и от преждевременных микросервисов.  
- **Корректность multi-tenant** — RLS + tenant context нужно доказывать негативными тестами, а не только happy path.  
- **Контроль scope** — строить платформу *и* первый вертикальный продукт, не замораживая ни то ни другое.  
- **Операционная реальность** — метрики, GHCR-релизы и compose-стеки появились раньше полного staging-автодеплоя и drill’ов восстановления бэкапов.

## 9. Чем гордимся

- Рабочий **каркас kernel**: identity, сессии, RBAC, tenancy, commands, events, audit  
- Настоящие **доменные модули**, а не hello-world CRUD: finance (двойная запись), campus, tasks, files, notifications, terminal  
- **AI-путь** с секретами и tenant identity на сервере, бюджетами и мультипровайдером  
- **OpenAPI + problem+json** как публичный контракт  
- **Базовая observability**: structured logs, request_id между сервисами, Prometheus `/metrics`  
- **Документация, которая учит**: architecture guides, ADR, «как писать модуль», learn-track  
- **Воспроизводимый стек**: `make stack` / Compose — Postgres + nexd + web + ai-gateway  

---

## 10. Чему научились

- AI-фичи разваливаются без платформенных примитивов (authz, tenancy, audit, budgets).  
- Модульный монолит — это продуктовая стратегия, а не только раскладка папок: он решает, как быстро выйдет следующий домен.  
- Красивый фронт без дисциплины write-path даёт демо, а не систему.  
- Документация и learning-материалы как часть продукта делают репозиторий проходимым для новых людей.  
- «Какой провайдер» — забота gateway; «кому можно спрашивать что» — забота kernel. Смешивать их — способ получить утечки.

## 11. Что дальше

Ближайшее:

- Дожать первый college-вертикаль end-to-end (меньше моков, больше живых API-экранов)  
- Продолжить модернизацию фронта против OpenAPI-контракта  
- Автоматизация staging, бэкапы и проверка restore  
- Глубже observability (дашборды, алерты, полноценный tracing)  

Направление платформы:

- Новые доменные модули (расписание, глубина оценок, приёмная кампания)  
- Действия ИИ через ту же command bus (agent как актор «по-настоящему»)  
- Опционально OIDC, сильный поиск, пакеты экспорта  
- Пилот с реальным учреждением после закрытия security-чеклиста и ops-пути
