# internal/platform/httpapi/authz.go

Страж (guard) для читающих HTTP-маршрутов: проверяет, что у актора запроса есть нужное разрешение, прежде чем пустить его в хендлер. Мутирующие запросы (команды) авторизует отдельно шина команд — этот файл покрывает только GET-подобные маршруты модулей, которые к шине не обращаются.

## Ключевое

- `RequirePermission(w, r, g *authz.Guard, permission string) bool` — главная функция: если `g == nil`, требует хотя бы аутентифицированного актора (режим "проверка не подключена", например in-memory без политики); иначе вызывает `g.Require` и сама пишет problem-ответ при отказе (401 при `ErrUnauthenticated`, 403 при `ErrDenied`, 500 на прочих ошибках).
- `requireActor(w, r) bool` — вспомогательная проверка: пропускает, если в контексте вообще есть аутентифицированный актор, без проверки конкретного права.

## Как это работает

Хендлеры модулей вызывают `RequirePermission(w, r, guard, "finance.accounts.read")` в начале своей логики и, если функция вернула `false`, сразу выходят — ответ об ошибке уже написан внутри. Собственно проверка права делегирована `authz.Guard.Require`, который сверяется с RBAC-политикой роль→разрешение; этот файл лишь транслирует её ошибки в HTTP problem-ответы нужного статуса.

## Связи

Зависит от `internal/kernel/authz` (`Guard`, `ErrUnauthenticated`, `ErrDenied`) и `internal/kernel/identity` (`ActorFrom`). Использует `WriteProblem` из problem.go. Вызывается из HTTP-хендлеров read-маршрутов различных модулей (`internal/module/*`), которые монтируются через `RouterConfig.Mount`.

## На что обратить внимание

`RequirePermission` пишет ответ в `w` самостоятельно — вызывающий код не должен дублировать запись ошибки, только проверить возвращаемое `bool` и сделать `return`. `g == nil` — это не ошибка конфигурации, а поддерживаемый режим работы (например, ранние стадии разработки без подключённой RBAC-политики).
