# internal/platform/httpapi/idempotency.go

Middleware, реализующий идемпотентность мутирующих запросов через заголовок `Idempotency-Key`. Нужен для офлайн-клиентов и сетевых ретраев: если запрос с уже виденным ключом придёт повторно, вместо повторного выполнения возвращается сохранённый ранее ответ.

## Ключевое

- `IdemState` — состояние ключа: `Fresh` (новый, надо исполнять), `InProgress` (тот же ключ выполняется параллельно другим запросом), `Done` (уже завершён — есть сохранённый `Status`/`ContentType`/`Body`).
- `IdempotencyStore` — интерфейс хранилища ключей: `Begin` (начать/проверить ключ), `Complete` (сохранить успешный результат), `Forget` (забыть ключ — для ретраев после сбоя). Реализация — `platform/postgres`, чтобы состояние переживало рестарт процесса.
- `idempotency(store IdempotencyStore) middleware` — сама middleware-функция.
- `bufferingRecorder` — обёртка над `http.ResponseWriter`, которая одновременно пропускает ответ клиенту и копит его в буфер (до `idemMaxBody` = 1 МБ) для сохранения в `store`.
- `idemMaxBody = 1 << 20` — предел размера ответа, который вообще сохраняется для повтора.
- `isMutating(method string) bool` — проверяет, что метод POST/PUT/PATCH/DELETE.

## Как это работает

Middleware активируется только для мутирующих запросов с непустым и не слишком длинным (`≤200` символов) заголовком `Idempotency-Key`, и только когда в контексте уже есть tenant (иначе ключ негде хранить в разрезе арендатора). Вызывается `store.Begin`: если ключ уже `Done` — отдаётся сохранённый ответ с пометкой заголовка `Idempotency-Replayed: true`; если `InProgress` — клиент получает 409 Conflict с просьбой повторить позже; если `Fresh` — запрос реально исполняется через `bufferingRecorder`, который параллельно копит тело ответа. По завершении: ответы 5xx или превысившие лимит размера тела НЕ кэшируются (`store.Forget`) — считается, что клиент вправе честно повторить попытку; остальные сохраняются через `store.Complete`.

## Связи

Зависит от `internal/kernel/tenancy` (`TenantFrom`) для проверки наличия tenant'а в контексте. Использует `WriteProblem` из problem.go. Подключается в routes.go как один из внутренних слоёв цепочки middleware (`cfg.Idempotency != nil`) — специально после разрешения tenant'а (`tenantResolver`), поскольку ключ хранится в разрезе окончательного tenant'а.

## На что обратить внимание

Ключевой инвариант: успех (`Complete`) сохраняется, а сбой (`Forget`) — нет. Это значит, что при временной ошибке сервера (5xx) клиент может безопасно повторить тот же запрос с тем же ключом и получить честное новое исполнение, а не застрять на сохранённой ошибке. `InProgress` (409) защищает от гонки, когда клиент отправляет повтор до завершения первого запроса (например, из-за таймаута на клиенте) — не считается ошибкой самого клиента, а сигналом "подождите и повторите".
