# internal/platform/httpapi/requestid.go

Middleware, который присваивает каждому HTTP-запросу уникальный идентификатор (request ID) и делает его доступным по всему пути обработки запроса — от логов до аудита и ответа клиенту. Это связующее звено для трассировки: по одному значению можно найти всё, что произошло с конкретным запросом.

## Ключевое

- `requestIDHeader` — имя HTTP-заголовка (`X-Request-Id`), в котором ID приходит от клиента/прокси и возвращается в ответе.
- `RequestIDFrom(ctx)` — экспортируемая функция, достаёт ID запроса из контекста; возвращает пустую строку, если middleware не отработал.
- `requestID()` — сам middleware: проверяет заголовок запроса, при отсутствии или неразумной длине генерирует новый ID, кладёт его в контекст и в заголовок ответа.
- `newRequestID()` — генерирует 128-битный случайный идентификатор в hex-формате через `crypto/rand`.

## Как это работает

Middleware либо доверяет ID, присланному клиентом/прокси (если он не пустой и не длиннее 64 символов), либо генерирует новый случайный. ID кладётся в `context.Context` через неэкспортируемый ключ типа `requestIDKey{}` (защита от коллизий с ключами других пакетов) и одновременно пишется в заголовок ответа, чтобы клиент тоже мог его увидеть. Дальше по цепочке обработки любой код с доступом к `context.Context` запроса может достать этот ID через `RequestIDFrom`.

## Связи

Подключается как самый внешний слой в middleware-цепочке в `routes.go` (`NewRouter`) — так, чтобы ID существовал ещё до логирования и записи в аудит. Используется `logging`-слоем (структурные логи включают ID) и `postgres.AuditRecorder` (поле `traceID` — обычно и есть `httpapi.RequestIDFrom`), что связывает HTTP-запрос, лог-строки и запись в журнале аудита одним идентификатором.

## На что обратить внимание

Отказ `crypto/rand.Read` в `newRequestID` не приводит к панике или ошибке запроса — возвращается заглушка `"0000000000000000"`. Это осознанный компромисс: телеметрия не должна ронять реальные запросы.
