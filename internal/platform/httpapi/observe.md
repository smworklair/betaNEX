# internal/platform/httpapi/observe.go

Middleware наблюдаемости: собирает метрики по каждому запросу (маршрут, статус, длительность) и опционально монтирует профилировщик `pprof` для development-окружения.

## Ключевое

- `observer(mux *http.ServeMux, observe func(route string, status int, dur time.Duration)) middleware` — middleware метрик, вызывающая переданный колбэк `observe` по завершении каждого запроса.
- `mountPprof(mux *http.ServeMux)` — регистрирует стандартные `/debug/pprof/*` эндпоинты (`Index`, `Cmdline`, `Profile`, `Symbol`, `Trace`) из `net/http/pprof`.

## Как это работает

`observer` оборачивает запрос `statusRecorder`'ом (из middleware.go), чтобы узнать итоговый статус, и после выполнения хендлера определяет шаблон маршрута через `mux.Handler(r)` — этот метод `ServeMux` сопоставляет запрос с зарегистрированным паттерном без реального исполнения хендлера. Благодаря этому в метку метрики попадает нормализованный путь вида `"GET /api/v1/finance/accounts"`, а не сырой URL с конкретными ID — это не даёт кардинальности метрик расти вместе с объёмом данных. Немаршрутизированные запросы помечаются как `"unmatched"`.

## Связи

Зависит от `net/http`, `net/http/pprof`, `time`; использует `statusRecorder` из middleware.go. Подключается в routes.go: `observer` ставится самым внешним слоем среди "обычных" middleware (перед `requestID`/`recoverer`), чтобы честно учитывать 500, порождённые `recoverer`, как реальный результат запроса. `mountPprof` включается только при `cfg.Pprof` (development) — комментарий явно указывает, что профилировщик не должен быть доступен в проде.

## На что обратить внимание

Вызов `mux.Handler(r)` внутри `observer` — не бесплатный трюк, а особенность API `http.ServeMux`: он позволяет заранее узнать, какой паттерн подойдёт к запросу, не выполняя сам хендлер повторно (реальное выполнение делает `next.ServeHTTP` до этого вызова). Порядок наблюдения важен: метрика видит статус уже после того, как `recoverer` (ниже по цепочке) успел превратить панику в 500.
