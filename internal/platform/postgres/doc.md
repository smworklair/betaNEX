# internal/platform/postgres/doc.go

Go-doc пакета `postgres` — точка входа для понимания всего адаптера платформы к PostgreSQL. Файл не содержит кода, кроме объявления пакета с комментарием, описывающим состав пакета.

## Ключевое

- Пакетный комментарий описывает пять частей пакета: `Connect`/`Ready` (пул `pgx` и readiness-пинг для `/readyz`), `InTenantTx`/`InTx` (транзакции с установкой `SET LOCAL app.tenant_id`), `Migrate` (применение goose-миграций, встроенных в бинарник), `ResolveTenant`/`EnsureTenant`/`CreateTenant` (реестр tenant'ов) и каталог `queries/` с SQL для генерации sqlc-кода в `db/`.

## Как это работает

`InTenantTx`/`InTx` — центральный механизм мультиарендности на уровне БД: перед выполнением запросов в транзакции выставляется `SET LOCAL app.tenant_id` из tenant-контекста запроса (`internal/kernel/tenancy`). Row-Level Security политики Postgres читают эту сессионную переменную и автоматически ограничивают видимые строки текущим tenant'ом — поэтому запросы, выполненные вне такой транзакции (или с неверно установленным tenant'ом), просто не вернут данных, без явной ошибки доступа.

## Связи

`postgres` — это пакет, от которого зависят почти все остальные файлы platform/postgres (`audit.go`, `auditread.go`, `authstore.go`, `idempotency.go`) и внешние пакеты вроде `outbox`. Сам пакет опирается на `internal/kernel/tenancy` (tenant-контекст) и на сгенерированный sqlc-код в `internal/platform/postgres/db`. `queries/*.sql` — источник для `make sqlc`, регенерирующей типобезопасный доступ к БД.

## На что обратить внимание

RLS + `SET LOCAL app.tenant_id` — это архитектурный инвариант всей мультиарендности NEX, а не деталь одного файла: любой новый код, читающий/пишущий tenant-специфичные таблицы, обязан идти через `InTenantTx`/`InTx`, иначе тихо не увидит данных (или, что хуже, для не-FORCE-RLS таблиц может увидеть чужие).
