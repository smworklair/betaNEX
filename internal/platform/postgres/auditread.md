# internal/platform/postgres/auditread.go

Реализация чтения журнала аудита из Postgres — обратная сторона `audit.go`: отдаёт записи (кто и что менял) для UI-вьюера аудита текущего tenant'а.

## Ключевое

- `AuditReader` — реализация интерфейса `audit.Reader` поверх `postgres.DB`.
- `NewAuditReader(d) *AuditReader` — конструктор.
- `(*AuditReader) Entries(ctx, f audit.Filter) ([]audit.Entry, error)` — возвращает записи журнала, свежие первыми, с фильтрами по команде и актору.

## Как это работает

`Entries` нормализует лимит выборки (по умолчанию 100, максимум 500), опционально добавляет фильтры `Command`/`ActorID` в параметры SQL-запроса и выполняет его через `db.InTenantTx` — транзакцию с установленным tenant'ом текущего контекста. Отдельного фильтра по tenant'у в Go-коде нет: за это отвечает Row-Level Security на уровне Postgres, которая сама ограничивает видимые строки транзакцией. Каждая строка результата конвертируется в доменный тип `audit.Entry`; поле `Diff` (jsonb) распаковывается из JSON, и если это не удаётся — ошибка игнорируется, а не валит всю выборку (журнал ценнее одной битой записи).

## Связи

Реализует интерфейс `audit.Reader` из `internal/kernel/audit` (`var _ audit.Reader = (*AuditReader)(nil)`). Зависит от `internal/platform/postgres/db` (сгенерированный sqlc-код, `Queries.ListAuditEntriesFiltered`) и от `postgres.DB.InTenantTx`. Используется, вероятно, HTTP-эндпоинтом просмотра журнала аудита (в модулях или в `httpapi`) — через интерфейс `audit.Reader`, без прямой зависимости от Postgres.

## На что обратить внимание

Изоляция по tenant'у здесь целиком лежит на RLS-политике БД, а не на явном WHERE-условии в Go-коде — это общий для платформы паттерн (см. также `doc.go`), но здесь он особенно важен: без активной tenant-транзакции запрос физически не увидит ни одной строки, даже без явной ошибки.
