# internal/platform/postgres/postgres.go

Точка входа платформенного пакета `postgres`: определяет тип `DB` — обёртку над пулом соединений pgx, которую создаёт композиционный корень приложения один раз при старте и передаёт во все модули.

## Ключевое

- `DB` — структура с единственным полем `pool *pgxpool.Pool`; вся работа с Postgres в NEX идёт через методы этого типа (здесь и в соседних файлах пакета — `tx.go`, `tenants.go`, `maintenance.go`).
- `Connect(ctx, dsn) (*DB, error)` — парсит DSN, создаёт пул и сразу пингует его с таймаутом, чтобы ошибка конфигурации (неверный DSN, недоступная БД) проявилась при старте процесса, а не на первом пользовательском запросе.
- `(*DB) Close()` — закрывает пул при остановке процесса.
- `(*DB) Pool() *pgxpool.Pool` — отдаёт «голый» пул для случаев, где нужен прямой доступ (тесты, будущие фоновые задачи River).
- `(*DB) Ready(ctx) error` — пинг для readiness-проверки (`/readyz`).
- `connectTimeout` — таймаут первого пинга (5 секунд).

## Как это работает

`Connect` — единственный способ получить `*DB` в проекте: парсинг DSN и первый пинг сделаны здесь, чтобы обвязка (логирование, retry, readiness) была в одном месте. Дальше все SQL-запросы модулей идут не через `Pool()` напрямую, а через транзакционные обёртки из `tx.go`, потому что только они устанавливают tenant-контекст, необходимый для RLS-политик Postgres.

## Связи

Использует `github.com/jackc/pgx/v5/pgxpool`. От этого файла зависят все остальные файлы пакета `postgres` (`tx.go`, `tenants.go`, `maintenance.go`, `migrate.go` косвенно) и композиционный корень (`cmd/nexd`), который вызывает `Connect` при старте.

## На что обратить внимание

Комментарий к `Pool()` прямо предупреждает: прикладные запросы должны идти через `InTenantTx` (см. `tx.go`), а не через `Pool()` напрямую — иначе RLS-политики не увидят `app.tenant_id` и просто не вернут ни одной строки (тихий отказ, не ошибка).
