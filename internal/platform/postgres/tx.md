# internal/platform/postgres/tx.go

Сердце мультиарендности NEX на уровне БД: обёртки над транзакциями Postgres, которые перед выполнением запроса устанавливают `app.tenant_id` в текущей сессии — именно на это значение опираются RLS-политики (row-level security), которые физически не дают одному tenant'у увидеть данные другого.

## Ключевое

- `ErrNoTenant` — в контексте нет tenant'а, а операция его требует.
- `ErrInvalidTenant` — идентификатор tenant'а в контексте не UUID.
- `txKey` — неэкспортируемый ключ контекста для уже открытой транзакции, чтобы вложенные вызовы могли к ней присоединиться, а не открывать новую.
- `(*DB) InTenantTx(ctx, fn) error` — исполняет `fn` в транзакции с обязательным tenant-контекстом; без tenant'а в `ctx` — ошибка `ErrNoTenant`. Это основной способ выполнить любой прикладной SQL-запрос в NEX.
- `(*DB) InTx(ctx, fn) error` — как `InTenantTx`, но не требует tenant'а (устанавливает его, если он всё же есть). Для системных операций (например, запись аудита отклонённого запроса).
- `(*DB) RunTx(ctx, fn) error` — реализует интерфейс `command.TxRunner` для шины команд: тот же `InTx`, но с сигнатурой без `*db.Queries`.

## Как это работает

`inTx` — общая реализация: если транзакция уже открыта выше по стеку (например, шиной команд, которая оборачивает и хендлер, и запись аудита в одну транзакцию), `fn` присоединяется к ней через `txFrom(ctx)`, не открывая новую. Иначе открывается новая транзакция, для tenant-варианта выполняется `SELECT set_config('app.tenant_id', $1, true)` (параметризованный запрос — исключает SQL-инъекцию через сам идентификатор), и это значение действует до конца транзакции благодаря `SET LOCAL`-семантике третьего аргумента `true`. При успехе (`fn` вернула `nil`) — коммит, при ошибке — откат через `defer tx.Rollback` (no-op после успешного коммита).

## Связи

Использует `internal/kernel/tenancy` (`TenantFrom`) для чтения tenant'а из контекста и `internal/platform/postgres/db` (сгенерированный sqlc-код) для типа `*db.Queries`. Тип `DB` определён в `postgres.go`. Практически все модули (`internal/module/*`) вызывают `InTenantTx` в своих репозиториях; шина команд использует `RunTx`.

## На что обратить внимание

Это единственный правильный путь к БД для прикладного кода: запрос через «голый» `Pool()` не имеет `app.tenant_id`, и RLS-политики Postgres просто не вернут ни одной строки — не ошибка, а тихий пустой результат, который легко принять за баг в другом месте. `InTx` специально сделан «безопасным по умолчанию»: если tenant в контексте есть, он всё равно установится, поэтому системный код может использовать `InTx` и для tenant-сценариев, и для действительно системных.
