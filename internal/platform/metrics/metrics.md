# internal/platform/metrics/metrics.go

Самописная минимальная реализация метрик в текстовом формате Prometheus — без зависимости от `client_golang`. Даёт счётчики (`Counter`), гистограммы (`Histogram`) и вычисляемые в момент скрейпа значения (`GaugeFunc`), плюс HTTP-хендлер `/metrics`.

## Ключевое

- `DefBuckets` — стандартные границы гистограммы латентности (как у Prometheus по умолчанию).
- `Registry` — потокобезопасный (`sync.Mutex`) реестр всех метрик процесса; хранит порядок регистрации для стабильного вывода.
- `New() *Registry` — конструктор пустого реестра.
- `(*Registry) Counter(name, help, labels...)` / `Histogram(name, help, buckets, labels...)` — идемпотентная регистрация метрики (повторный вызов с тем же именем — no-op).
- `(*Registry) GaugeFunc(name, fn)` — регистрирует показатель, значение которого вычисляется прямо в момент запроса `/metrics` (например, `pgxpool.Stat`, число горутин).
- `(*Registry) Inc(name, labelVals...)` / `Observe(name, v, labelVals...)` / `ObserveDuration(name, d, labelVals...)` — запись значений; обращение к незарегистрированной метрике — тихий no-op, не паника.
- `(*Registry) Handler() http.Handler` — отдаёт все метрики в текстовом формате Prometheus 0.0.4.
- `(*Registry) RegisterRuntime()` — добавляет базовые gauge-метрики рантайма Go (`nex_goroutines`, `nex_heap_alloc_bytes`).

## Как это работает

Каждая метрика — это `family` (тип: counter/histogram, набор допустимых меток), а внутри неё — набор `row` по каждой уникальной комбинации значений меток (ключ — join значений через `\x00`). Инкремент/наблюдение ищут (или создают) нужную `row` и обновляют счётчики. При отдаче через `Handler` сначала выводятся gauge'ы (вызывается функция прямо сейчас), затем для каждой family — HELP/TYPE-заголовки и отсортированные по ключу строки данных в формате, который понимает Prometheus text exposition format (для гистограмм — кумулятивные `_bucket`, плюс `_sum` и `_count`).

## Связи

Не зависит от других пакетов NEX — самодостаточный примитив платформы. Реестр создаётся в композиционном корне (`cmd/nexd`) и передаётся туда, где нужно инкрементировать метрики: в `httpapi` через `RouterConfig.Observe` (см. `routes.go`) для метрик по маршрутам, и в другие места платформы (например, `postgres`, `outbox`) для метрик доменных операций.

## На что обратить внимание

Реестр специально спроектирован так, чтобы не требовать внешней зависимости `client_golang`: комментарий пакета прямо говорит, что при росте требований (exemplars, native histograms) можно будет мигрировать без изменения интерфейса вызывающего кода. Обращение к несуществующей метрике (`Inc`/`Observe`) — намеренно тихая операция, а не паника: ошибка в имени метрики не должна ронять обработку запроса.
