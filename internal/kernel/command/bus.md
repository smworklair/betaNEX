# internal/kernel/command/bus.go

Конкретная реализация шины команд — `MemoryBus`, единственный путь изменения данных в NEX. Проводит каждую команду через цикл: валидация → авторизация → исполнение (в транзакции, если доступна) → фиксация исхода в аудите.

## Ключевое

- `HandlerFunc` — функция-обработчик конкретной команды; регистрируется модулями при старте приложения.
- `Authorizer` — интерфейс проверки прав актора на исполнение команды (реализация — `authz.PolicyAuthorizer`; интерфейс объявлен здесь, чтобы `command` не зависел от `authz` — зависимости направлены внутрь).
- `TxRunner` — интерфейс запуска функции в транзакции хранилища (реализация — `platform/postgres`); тоже объявлен здесь ради инверсии зависимостей.
- `ErrUnknownCommand`, `ErrAlreadyRegistered` — ошибки шины, проверяемые через `errors.Is`.
- `MemoryBus` — сама шина: держит `Authorizer`, `audit.Recorder`, опциональный `TxRunner` и потокобезопасную карту зарегистрированных хендлеров (`sync.RWMutex`).
- `Option`, `WithTxRunner(tx)` — функциональная опция, включающая транзакционное исполнение.
- `NewMemoryBus(authz, rec, opts...) *MemoryBus` — конструктор.
- `(*MemoryBus) Register(name string, h HandlerFunc) error` — регистрирует хендлер под именем команды; повторная регистрация — ошибка.
- `(*MemoryBus) Dispatch(ctx, cmd Command) error` — главный метод: полный жизненный цикл команды.
- `(*MemoryBus) execute`, `record`, `recordErr` — внутренние шаги: исполнение хендлера (в транзакции при наличии `TxRunner`), фиксация записи в аудите best-effort для отказов/ошибок и строго для успеха.

## Как это работает

`Dispatch` сначала ищет зарегистрированный хендлер по `cmd.Name()` — если его нет, сразу пишет `OutcomeError` в аудит и возвращает `ErrUnknownCommand`. Затем вызывает `cmd.Validate()` (дешёвая проверка инвариантов без обращения к БД); при ошибке — аудит с `OutcomeError`. Дальше идёт `authz.Authorize(ctx, cmd)`; отказ пишется как `OutcomeDenied`. Если обе проверки пройдены, вызывается `execute`, который создаёт коллектор диффа (`audit.WithCollector`), запускает хендлер и, если он не вернул ошибку, тут же пишет запись `OutcomeOK` — и всё это (исполнение + запись успеха) оборачивается в `tx.RunTx`, если задан `TxRunner`: таким образом хендлер и запись аудита об успехе физически находятся в одной транзакции БД. Если хендлер вернул ошибку, она фиксируется в аудите отдельной (уже не транзакционной) записью после отката — иначе запись об ошибке откатилась бы вместе с изменениями.

## Связи

Зависит от `internal/kernel/audit` (типы `Entry`, `Recorder`, `Outcome`, функции `WithCollector`/`CollectedDiff`), `internal/kernel/identity` (`ActorFrom` для ID актора в записи), `internal/kernel/tenancy` (`TenantFrom` для ID tenant'а). Реализует интерфейс `Bus`, объявленный в `command.go` того же пакета. Создаётся в `cmd/nexd/main.go` (`command.NewMemoryBus(authz.NewPolicyAuthorizer(policy), recorder, busOpts...)`), где `busOpts` включает `command.WithTxRunner(pg)` при наличии Postgres. Все доменные модули (`internal/module/*`) вызывают `bus.Register(...)` при старте и `bus.Dispatch(...)` из HTTP-хендлеров.

## На что обратить внимание

Ключевой инвариант: успех записывается в ту же транзакцию, что и изменение данных (`RunTx` оборачивает и хендлер, и запись аудита), а вот отказ авторизации и ошибка исполнения пишутся уже вне транзакции, best-effort, отдельным вызовом `record` — потому что откатившаяся транзакция стёрла бы такую запись вместе с несостоявшимся изменением. Это асимметрия по дизайну, а не недосмотр.
