# internal/kernel/tenancy/context.go

Небольшой утилитарный файл, реализующий перенос идентификатора tenant'а (организации) через `context.Context` — базовый механизм мультиарендности NEX. На эти две функции опирается вся цепочка от HTTP-запроса до SQL-запроса.

## Ключевое

- `tenantKey` — неэкспортируемый пустой тип-ключ для `context.Value`, гарантирует, что снаружи пакета никто не подложит и не подсмотрит значение напрямую, только через функции ниже.
- `WithTenant(ctx, id) context.Context` — кладёт ID tenant'а в контекст; вызывается HTTP middleware после того, как определена организация запроса.
- `TenantFrom(ctx) (string, bool)` — достаёт ID tenant'а из контекста; `false` во втором значении означает, что tenant не установлен, и запрос не имеет права трогать доменные данные.

## Как это работает

Middleware в HTTP-слое (не в этом файле) определяет tenant запроса (например, по поддомену, заголовку или сессии) и вызывает `WithTenant`, оборачивая контекст запроса. Дальше этот контекст передаётся по всей цепочке вызовов — в обработчики команд, в репозитории — и там уже `TenantFrom` используется, чтобы получить ID и передать его дальше в SQL (например, как параметр `SET LOCAL app.tenant_id`). Паттерн стандартный для Go: значение путешествует неявно через контекст, а не через явные аргументы функций.

## Связи

Используется практически всеми модулями (`internal/module/*`) в их репозиториях — например, `campus/pgrepo.go` и `files/pgrepo.go` вызывают `tenancy.TenantFrom(ctx)`, чтобы получить tenant для SQL-запроса. Также используется HTTP-обработчиками (`files/http.go`) для ранней проверки, что tenant вообще определён. Зависит только от стандартного пакета `context`.

## На что обратить внимание

Отсутствие tenant'а в контексте — не ошибка сама по себе на уровне этого файла (просто `false`), но модули трактуют это как жёсткую ошибку (`ErrNoTenant`) перед любым обращением к БД — то есть каждый модуль обязан сам проверить `ok` и отказать в обработке, иначе есть риск случайно выполнить запрос без изоляции по tenant'у.
