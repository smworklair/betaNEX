# internal/kernel/tenancy/doc.go

Doc-комментарий пакета `tenancy`, описывающий модель мультиарендности NEX — как данные разных организаций (tenant'ов) изолируются друг от друга внутри одной базы и одного приложения.

## Ключевое

- `package tenancy` — пакет ядра, отвечающий за изоляцию данных между организациями (tenant'ами).
- Модель: каждая доменная строка в БД несёт `tenant_id`.
- Два рубежа изоляции: (1) tenant-контекст, протаскиваемый из HTTP middleware через команды вплоть до SQL, где выставляется `SET LOCAL app.tenant_id` в каждой транзакции; (2) Row-Level Security (RLS) в Postgres как второй, страхующий рубеж.
- Жёсткое правило: ни один запрос к доменным данным не выполняется без tenant-контекста.

## Как это работает

Файл не содержит исполняемого кода, только doc-комментарий, задающий контракт для всего остального пакета (в частности, для `context.go`). Идея в defense-in-depth: даже если приложение по ошибке забудет отфильтровать по tenant_id в запросе, RLS на уровне Postgres не даст прочитать чужие данные — то есть изоляция не полагается только на дисциплину прикладного кода.

## Связи

Является основой для `internal/kernel/tenancy/context.go` (WithTenant/TenantFrom) и для инфраструктуры `internal/platform/postgres` (например, `InTenantTx`, которая, судя по коду репозиториев модулей, выставляет tenant в транзакции). От пакета зависят все доменные модули (campus, files, finance) — их репозитории на каждый SQL-вызов оборачивают операции через tenant-контекст.

## На что обратить внимание

Двухуровневая защита (приложение + RLS) — намеренное архитектурное решение, а не избыточность: приложенческая часть даёт удобство и производительность, RLS — гарантию на случай багов в бизнес-логике. При чтении кода модулей стоит помнить, что отсутствие явной проверки tenant_id в конкретном SQL-запросе не обязательно баг — RLS может подстраховывать на уровне БД.
