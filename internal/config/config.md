# internal/config/config.go

Единственное место в проекте, где читаются переменные окружения. Собирает их в неизменяемую структуру `Config`, которая затем явно передаётся всем компонентам системы (в основном — в `cmd/nexd/main.go`). Следует 12-factor принципу: вся конфигурация — в окружении, всё под префиксом `NEX_`.

## Ключевое

- `Environment` / `EnvDevelopment`, `EnvProduction` — тип и значения окружения развёртывания; используется только для дефолтов (например, формат логов), а не для ветвления доменной логики.
- `Config` — корневая структура: `Env`, `HTTP`, `DB`, `Auth`, `Files`, `Log`. Строится один раз функцией `Load` и больше не меняется.
- `HTTPConfig` — адрес, CORS origins, тайм-ауты HTTP-сервера (чтение, запись, простой, graceful shutdown).
- `DBConfig.URL` — строка подключения к Postgres; пустая строка = режим in-memory (без персистентности).
- `AuthConfig` — `SessionTTL` (время жизни сессии со скользящим продлением) и `CookieSameSite` (`lax`/`strict`/`none`).
- `FilesConfig` — каталог для файлов на диске и лимит размера загрузки.
- `LogConfig` — уровень и формат (`json`/`text`) логирования.
- `Load() (Config, error)` — читает переменные окружения, применяет дефолты, валидирует и возвращает `Config`. Копит все ошибки парсинга сразу, а не падает на первой.
- `(Config) validate() error` — проверяет согласованность уже собранных значений (допустимые enum-значения, непустой адрес, положительный TTL, корректный формат CORS origin и т.д.), используя `errors.Join`.
- `envReader` — приватный помощник для чтения типизированных значений (`str`, `list`, `duration`, `int64`) с накоплением ошибок парсинга.

## Как это работает

`Load` создаёт `envReader` и последовательно заполняет поля `Config`, читая переменные окружения с дефолтами по умолчанию. После этого применяются два вычисляемых правила: (1) формат лога по умолчанию — `json` в production и `text` в development, если явно не задан; (2) `CookieSameSite` по умолчанию — `none`, если заданы CORS-origins (кросс-доменный фронтенд), иначе `lax`. В конце вызываются `r.err()` (ошибки парсинга) и `cfg.validate()` (ошибки бизнес-правил) — если что-то не так, возвращается объединённая ошибка со всеми проблемами разом.

## Связи

Не зависит ни от чего внутри проекта — только от стандартной библиотеки. От него зависит `cmd/nexd/main.go` (единственный вызывающий `config.Load()`), который передаёт `Config` дальше во все компоненты (`httpapi`, `postgres`, `logging`, `auth` и т.д.).

## На что обратить внимание

Логика вывода `CookieSameSite=none` из наличия `CORSOrigins` — не случайность, а обход реального поведения браузеров: cookie с `SameSite=Lax` не отправляется при кросс-сайтовом fetch, поэтому кросс-доменный фронтенд (например, на Vercel) молча теряет сессию, если явно не выставить `None` (что также требует флага `Secure`).
