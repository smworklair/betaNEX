# internal/module/finance/doc.go

Doc-комментарий пакета `finance` — описывает архитектурную идею модуля «Финансы»: леджер двойной записи как единый примитив, на котором строится весь финансовый раздел системы (счета студентов, платежи, задолженности, бюджет).

## Ключевое

- `package finance` — модуль с классической моделью двойной записи (ссылка на ADR-021 — архитектурное решение зафиксировано отдельным документом).
- Идея переиспользования одного примитива: план счетов + сбалансированные проводки покрывают разные бизнес-сценарии колледжа — платёж студента («касса/банк → доход»), начисление платы («дебиторка студента → доход»), задолженность (сальдо лицевого счёта студента), бюджет (счета расходов).
- Правила модуля: деньги — `int64` в минорных единицах (копейках), никакого `float`; каждая проводка сбалансирована (дебет = кредит); проводки неизменяемы и не удаляются, исправления делаются только сторнирующей (обратной) проводкой — append-only модель, как и в аудит-логе; все изменения — через шину команд, чтение — через `Repository`.
- Указание на состояние хранилища на момент написания: работает `MemoryRepository` (данные в памяти, для разработки/тестов), Postgres-реализация на sqlc — на вехе M2 (черновики уже есть: `migrations/00002_finance.sql`, `internal/platform/postgres/queries/finance.sql`).

## Как это работает

Файл не содержит кода, только документацию, задающую контракт для остальных файлов пакета finance. Ключевая архитектурная мысль: вместо отдельных сущностей «платёж», «начисление», «задолженность» модуль моделирует всё через один универсальный механизм бухгалтерской проводки — это снижает число специализированных таблиц и бизнес-правил, но требует, чтобы разработчик, добавляющий новую финансовую операцию, думал в терминах дебета/кредита, а не заводил новую сущность.

## Связи

Задаёт контракт для `commands.go` (CreateAccount, PostEntry), `events.go` (EntryPosted), `handlers.go` (RegisterCommands), `export.go` (отчёты и экспорты) и для основного файла `finance.go` (не входит в этот список, но там определены `Account`, `Entry`, `Line`, `AccountType`, `Repository`, `PostgresRepository`, `MemoryRepository`).

## На что обратить внимание

Упоминание append-only (проводки не изменяются и не удаляются, только сторнируются) — важный инвариант: при чтении кода модуля не стоит искать методы вроде `UpdateEntry`/`DeleteEntry` — их не должно существовать в принципе, это осознанное решение, а не недоработка.
