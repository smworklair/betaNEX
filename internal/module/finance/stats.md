# internal/module/finance/stats.go

Отчётная витрина (materialized-style) месячных оборотов по счетам — надстройка над леджером для быстрой аналитики без пересчёта по всем проводкам «на лету». Определяет и команду ручного пересчёта, и методы доступа к витрине.

## Ключевое

- `PermStatsRefresh` — право `finance:stats:refresh`, нужное для пересчёта витрины.
- `CmdStatsRefresh` — стабильное имя команды `finance.stats.refresh` для аудита.
- `MonthlyTurnover` — строка витрины: обороты (дебет/кредит) конкретного счёта за конкретный месяц плюс отметка времени последнего пересчёта.
- `RefreshStats` — команда «пересчитать витрину»; данных не несёт (`struct{}`), реализует `Name()`/`Permission()`/`Validate()` для командной шины.
- `ErrStatsUnsupported` — ошибка «эта операция требует Postgres» — сигнализирует, что у `MemoryRepository` витрин нет.
- `(*PostgresRepository) RefreshStats(ctx) error` — сносит и заново заполняет витрину в одной транзакции.
- `(*PostgresRepository) MonthlyTurnovers(ctx) ([]MonthlyTurnover, error)` — читает витрину.

## Как это работает

`RefreshStats` реализован простым «снести и перелить»: `ClearFinanceMonthlyTurnovers` очищает таблицу-витрину, `FillFinanceMonthlyTurnovers` заполняет её заново одним SQL-проходом по всем проводкам. Комментарий поясняет, что для объёмов одного колледжа это операция на секунды; при росте данных её планируется заменить на инкрементальный пересчёт. Команда `RefreshStats` может прийти как через обычную командную шину (ручной или API-триггер), так и напрямую вызываться ночной cron-задачей мимо шины (через сам репозиторий) — то есть у неё два пути исполнения.

## Связи

Методы объявлены на `*PostgresRepository` из `pgrepo.go` — расширяют тот же тип. Использует `internal/platform/postgres/db` для sqlc-запросов. `ErrStatsUnsupported` подразумевает, что вызывающий код (HTTP-хендлеры или terminal-адаптеры) должен уметь работать без витрины, если под капотом `MemoryRepository`.

## На что обратить внимание

Команда `RefreshStats` не типизирует свою связь с cron-планировщиком — это часть контракта, зафиксированная только в комментарии («также запускается ночной cron-задачей напрямую через репозиторий»), а не в коде файла.
