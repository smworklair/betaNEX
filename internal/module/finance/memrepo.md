# internal/module/finance/memrepo.go

In-memory реализация `Repository` из `repo.go` для леджера finance. Используется в разработке и тестах, пока (или вместо) настоящей Postgres-реализации (`pgrepo.go`) — данные хранятся только в памяти процесса и теряются при рестарте.

## Ключевое

- `MemoryRepository` — потокобезопасное (через `sync.RWMutex`) хранилище счетов и проводок всех tenant'ов в памяти.
- `tenantLedger` — внутренняя структура данных одного tenant'а: карта счетов по ID, индекс кода счёта → ID (для проверки уникальности) и срез проводок.
- `NewMemoryRepository()` — конструктор, создаёт пустое хранилище.
- Методы `CreateAccount`, `Account`, `Accounts`, `PostEntry`, `Entries` — реализация интерфейса `Repository`.

## Как это работает

Каждый tenant получает свой `tenantLedger`, создаваемый лениво при первом обращении (метод `ledger`) — карта `tenants map[string]*tenantLedger` имитирует будущую изоляцию по `tenant_id` и RLS в Postgres. Сальдо счетов не хранится отдельно, а каждый раз пересчитывается в `Accounts()` полным проходом по всем проводкам: для каждой строки проводки берётся знак через `signFor` (из `ledger.go`) и накапливается в карте `sums`. `PostEntry` перед сохранением проверяет, что все счета строк существуют и что валюта у всех строк одинаковая — балансировку дебета/кредита проверяет ещё раньше `PostEntry.Validate()` в `commands.go`. Все операции берут единый мьютекс `mu` (чтение — `RLock`, запись — `Lock`).

## Связи

Реализует интерфейс `Repository` из `repo.go`, использует типы `Account`, `Entry`, `Line`, `signFor` из `ledger.go`, а также `internal/kernel/tenancy` — для извлечения tenant'а из контекста. Конкурирует по роли с `pgrepo.go` (та же реализация интерфейса, но поверх Postgres). Composition root (`cmd/nexd`) выбирает, какую из двух реализаций подключить к `http.go`.

## На что обратить внимание

Пересчёт сальдо полным проходом по всем проводкам при каждом вызове `Accounts()` — это осознанно упрощённое решение «для объёмов разработки»: комментарий в коде прямо говорит, что в Postgres это станет одним агрегирующим SQL-запросом. Не стоит переносить этот O(n) алгоритм как образец для production-кода.
