# internal/module/terminal/terminal.go

Ядро модуля «Терминал» — «Администратор · альфа»: текстовая консоль поверх всей системы, задуманная как AI-native вход во всю функциональность. Это не отдельная подсистема и не обход архитектуры: чтения идут через те же репозитории, что и обычные экраны, мутации — только через шину команд (авторизация + аудит + транзакция). Терминал — ещё один actor поверх того же командного «позвоночника», а не параллельный путь к данным.

## Ключевое

- `PermExec` — право `terminal:exec`, выдаётся только admin: терминал открывает всю систему разом, поэтому это отдельное право, а не сумма прав отдельных модулей.
- `Result` — единый структурированный ответ любой команды: `Kind` (`text` | `table` | `kpi`), `Title`, `Text`, `Columns`/`Rows` (для таблиц), `KPIs` (для сводок), `Hint` (подсказка следующего шага). Терминал отдаёт не готовый текст, а данные — фронт сам решает, как их отрисовать.
- `KPI{Label, Value}` — один показатель в ответе вида `kind=kpi`.
- Строчные DTO для консольного вывода: `TaskRow`, `UserRow`, `AuditRow`, `GroupRow`, `StudentRow`, `GradeRow`, `BalanceRow`, `EntryRow` — терминальные проекции сущностей из других модулей (задачи, пользователи, аудит, кампус, финансы).
- `Deps` — набор полей-функций (не интерфейсов на чужие типы!), через которые терминал получает данные и мутации от остальной системы: `Tasks`, `AddTask`, `DoneTask`, `Users`, `Notify`, `Audit`, `Unread`, `Groups`, `Students`, `Grades`, `Balances`, `Entries`. Модуль terminal не импортирует соседние модули напрямую — все адаптеры собирает composition root (`cmd/nexd`), сохраняя правильное направление зависимостей. Любое nil-поле в `Deps` просто отключает соответствующую команду (терминал живёт и с частичной сборкой).
- `Notify{UserIDs, Title}` — команда шины «отправить уведомление пользователям» (в отличие от прямых чтений, идёт через `command.Bus` — с авторизацией, валидацией и аудитом как у любой другой мутации).
- `RegisterCommands(bus, sender)` — регистрирует обработчик `CmdNotify` в шине.
- `(d Deps) Exec(ctx, line string) (Result, error)` — точка входа: разбирает строку и выполняет команду.
- `ErrUnknown` — команда не распознана; фронт в ответ показывает подсказку набрать `help`.

## Как парсится ввод

`Exec` разбивает строку на слова (`strings.Fields`) и смотрит на первое слово. Карта `canon` — таблица синонимов: терминал говорит с администратором по-русски («обзор», «долги», «люди»), но исторические английские написания («status», «users») тоже работают — оба варианта сводятся к одному каноническому имени команды (`"help"`, `"status"`, `"tasks"` и т.д.). Дальше `switch` по каноническому имени вызывает соответствующий приватный метод `Deps`. Некоторые команды имеют собственные под-варианты разбора аргументов: `task add <текст>` / `task done <id>` разбираются внутри case `"task"`, а их русские формы «новая задача <текст>» / «готово <id>» приходят как ОТДЕЛЬНЫЕ канонические имена (`task_add_ru`, `task_done_ru`) — то есть маппинг `canon` работает не 1:1 на команду, а различает разные словоформы одного и того же действия ещё на уровне первого слова. Свободный текст, не подпадающий ни под одну команду, в `Exec` вообще не попадает — фронт заранее отправляет его в LLM-ветку, а сюда идут только уже распознанные структурированные команды.

## Команды терминала

- `help` (`?`, «помощь») — таблица всех доступных команд со шпаргалкой.
- `whoami` («кто») — текущий актор и его роли, из `identity.ActorFrom(ctx)`.
- `status` («статус», «обзор», «сводка») — KPI-сводка: открытые задачи, число пользователей, непрочитанные уведомления текущего актора. Каждый источник опционален — что не подключено в `Deps`, то просто пропускается.
- `tasks` («задачи») `[open|done|all]` — таблица задач с фильтром по статусу (по умолчанию `open`).
- `task add <текст>` / `новая задача <текст>` — создать задачу (`d.AddTask`, идёт через шину на стороне composition root).
- `task done <id>` / `готово <id>` — закрыть задачу (`d.DoneTask`).
- `users` («люди», «пользователи») — таблица пользователей с ролями.
- `notify <email|all> <текст>` («уведомить») — резолвит email в список ID пользователей через `d.Users` и шлёт `d.Notify`; специальное значение `all` — разослать всем.
- `audit [n]` («аудит», «журнал») — последние `n` (по умолчанию 10) записей журнала аудита.
- `analytics` («аналитика») — сводный KPI по учебному контингенту: число групп, студентов, средний балл по последним оценкам.
- `groups` («группы») — таблица учебных групп.
- `students [поиск]` («студенты») — таблица студентов, опционально с поисковым запросом.
- `grades [n]` («оценки») — последние `n` (по умолчанию 15) записей учебного журнала.
- `finance` («финансы») — KPI: число счетов, сумма активов, сумма доходов, число проводок.
- `accounts` («счета») — таблица счетов с сальдо (форматированным в рубли через `rubKop`).
- `entries [n]` («проводки») — последние `n` (по умолчанию 15) проводок.
- `security` («безопасность») — KPI по безопасности: число пользователей, число админов (из `Users`), число отказов доступа среди последних 100 записей аудита (`Outcome == "denied"`).

## Как строится Result

Каждый приватный метод `Deps` (`status`, `tasks`, `finance` и т.д.) сначала проверяет, подключён ли нужный источник данных (`d.XXX == nil`) — если нет, возвращает `Result{Kind: "text", Text: "модуль ... не подключён"}` вместо ошибки, чтобы терминал оставался частично работоспособным при неполной сборке зависимостей. Затем данные из адаптера конвертируются в `Result`: списки построчно превращаются в `Rows` (`[][]string`) под заданными `Columns`, агрегаты — в срез `KPIs`. Пустой список обычно превращается в `Kind: "text"` с понятным сообщением («задач нет», «студентов пока нет») вместо пустой таблицы. Многие ответы дополнительно несут `Hint` — подсказку следующей команды (например, после `tasks` — «task done <id> — закрыть»).

## Связи

Зависит от `internal/kernel/command` (регистрация `Notify` в шине) и `internal/kernel/identity` (`ActorFrom` для `whoami`/`status`). Принципиально НЕ импортирует `internal/module/finance`, `tasks`, `notifications` и т.п. напрямую — вместо этого получает функции-адаптеры через `Deps`, которые собирает composition root (`cmd/nexd`), оборачивая настоящие репозитории и сервисы этих модулей. `http.go` того же пакета — единственный вызывающий код `Deps.Exec` снаружи.

## На что обратить внимание

`Deps` — не интерфейс, а структура из полей-функций: это осознанный архитектурный выбор, чтобы модуль terminal не приобретал зависимость на типы соседних модулей (иначе пришлось бы импортировать `finance.Repository`, `tasks.Repository` и т.д., что нарушило бы правило «модуль не знает соседей»). Каждое поле независимо и может быть `nil`.

`canon` — не биекция «слово → команда», а способ различать словоформы: например, «новая задача X» и «task add X» ведут к разным веткам `switch` (`task_add_ru` и `task`), хотя по смыслу это одна и та же операция — код для них частично дублируется (`addTask` вызывается из обеих веток), но сам разбор аргументов у них разный (у русской формы нужно ещё съесть слово «задача», если оно есть).

Мутации `AddTask`/`DoneTask`/`Notify` в комментариях к полям `Deps` помечены «через шину» — это значит, что настоящая авторизация и аудит происходят не в `terminal.go`, а внутри реализации этих функций-адаптеров на стороне composition root, которая внутри вызывает `bus.Dispatch`. Сам `terminal.go` этого не видно — важно не забывать при чтении кода.
