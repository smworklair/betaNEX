# internal/module/files/http.go

HTTP-слой модуля «Файлы»: маршруты загрузки, списка, скачивания и удаления файлов. Самый нетривиальный HTTP-файл среди рассмотренных модулей — здесь смешаны потоковая работа с диском и мутации через шину команд.

## Ключевое

- `Routes(bus, repo, store, maxUpload, guard) func(*http.ServeMux)` — монтирует `POST/GET /api/v1/files`, `GET/DELETE /api/v1/files/{id}`.
- `api` — структура-хендлер, держит шину, репозиторий, `blob.Storage`, лимит размера загрузки и guard.
- `fileDTO`/`dto(f File)` — форма ответа для JSON-эндпоинтов.
- `(*api).upload` — принимает multipart-запрос, стримит содержимое файла сразу в blob-хранилище, затем регистрирует метаданные командой `Attach` через шину.
- `(*api).list`, `(*api).download`, `(*api).remove` — чтение списка, отдача содержимого файла (с поддержкой Range-запросов через `http.ServeContent`), удаление.
- `writeErr` — маппинг ошибок модуля/blob в HTTP problem-ответы (403/400/404/500).
- `urlEscape` — ручное RFC 5987-кодирование имени файла для заголовка `Content-Disposition` (чтобы не ломались не-ASCII имена).

## Как это работает

При загрузке (`upload`) сначала проверяются tenant и актор в контексте, затем тело запроса читается как multipart-поток напрямую (`r.MultipartReader()`), part `file` сохраняется в blob-хранилище потоково (`h.store.Save`) без буферизации всего файла в память — это важно для больших файлов. Только после физического сохранения формируется команда `Attach` и отправляется через шину: так авторизация и аудит-запись гарантированно относятся к уже сохранённому байт-контенту. Если `Dispatch` команды падает (например, отказ прав), HTTP-слой проверяет через `repo.SHAReferenced`, не остался ли только что сохранённый блоб без ссылок, и подчищает его с диска — чтобы не копить мусор. Аналогичная логика работает при удалении (`remove`): сначала считывается SHA файла, затем идёт команда `Delete`, и только после успешного удаления метаданных проверяется, не осиротел ли блоб на диске.

## Связи

Зависит от `internal/kernel/authz` (Guard, ErrDenied), `internal/kernel/command` (Bus), `internal/kernel/identity` (ActorFrom), `internal/kernel/tenancy` (TenantFrom), `internal/platform/blob` (Storage, ErrNotFound) и `internal/platform/httpapi` (WriteJSON, WriteProblem, RequirePermission). Использует `*Repository` из `pgrepo.go` и команды/типы из `files.go`. Подключается композиционным корнем аналогично `campus.Routes`.

## На что обратить внимание

Порядок операций в `upload` и `remove` продуман против гонок и утечек: контент сохраняется/читается до транзакции с метаданными, а очистка блоба — после операции с метаданными, с проверкой оставшихся ссылок по SHA (дедупликация по содержимому — несколько файлов с одинаковым SHA256 могут ссылаться на один и тот же блоб на диске, поэтому удалить блоб можно только когда последняя ссылка исчезла).
