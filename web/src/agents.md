# web/src/agents.ts

Модель данных для будущего слоя «фоновых агентов» NEX — прототип идеи, что NEX не просто чат-ассистент, а набор автономных агентов по доменам (студенты, финансы, безопасность, расписание и т.д.), каждый со своим уровнем самостоятельности. На бэкенде этого модуля пока нет — файл целиком состоит из моков и вспомогательной логики над ними, которую использует `web/src/api/agents.ts`.

## Ключевое

- `AUTONOMY_LEVELS`, `type Autonomy = 0 | 1 | 2 | 3` — шкала автономии агента: 0 «Наблюдает» → 1 «Советует» → 2 «Готовит» (ждёт подтверждения) → 3 «Автономен» (делает сам, всё в аудите).
- `interface Agent` и `AGENTS: Agent[]` — семь агентов-моков (Куратор-риск, Финконтролёр, Страж, Диспетчер, Приёмная, Документовед, Кампус-скаут) с текущим уровнем автономии и последним действием.
- `interface PendingAction` и `QUEUE` — очередь действий агентов, ожидающих подтверждения человеком, с уровнем риска (`low`/`medium`/`high`).
- `interface AgentLogEntry` и `AGENT_LOG` — журнал уже выполненных агентами действий.
- `agentById(id)` — поиск агента по id (не проверяет отсутствие — падает, если id не найден).
- `AUTOPILOT_PRESETS` — три пресета («Осторожный»/«Стандартный»/«Смелый»), которые выставляют уровень автономии сразу всем агентам.
- `guessAgent(ruleText)` — подбирает подходящего агента под новое текстовое правило по ключевым словам (заглушка того, что на бэкенде должна делать LLM-классификация).
- `RULES`, `dryRun(action)` — список автоматизаций на естественном языке и текстовый прогноз последствий действия до подтверждения (dry-run), тоже заглушки под будущую LLM/симуляцию на сервере.

## Как это работает

Все данные — статические константы в памяти фронтенда, ничего не персистится и не ходит в сеть. Функции `guessAgent` и `dryRun` — простые regex-эвристики по русским ключевым словам, явно помеченные комментариями `BACKEND:` как временная замена того, что в реальной системе должна делать модель (классификация правила, симуляция действия).

## Связи

Потребляется через `web/src/api/agents.ts`, который реэкспортирует эти сиды как «API» (`listAgents`, `agentQueue`, `agentLog`), чтобы экраны раздела «Агенты» (`web/src/pages/agents`) работали через единый слой `api/`, не завязываясь напрямую на моки. Когда появится реальный бэкенд-модуль агентов, поменяется только `api/agents.ts`, не этот файл и не экраны.

## На что обратить внимание

Это единственный файл в списке задачи, который целиком описывает функциональность, отсутствующую на бэкенде, — сигнатуры типов здесь фактически являются черновиком будущего API-контракта, а не просто UI-моделью.
