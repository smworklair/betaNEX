# web/src/api/auth.ts

Обёртка над `/api/v1/auth/*` — вход, получение текущей сессии и выход. Единственный модуль `api/`, который отвечает за аутентификацию, и единственное место в фронтенде, откуда напрямую видно, что сессия хранится в httpOnly-cookie, а не в токене на клиенте.

## Ключевое

- `interface SessionUser` — форма ответа `/auth/me` и `/auth/login`: `id`, `email`, `display_name`, `roles`, `tenant`.
- `authConfigured(): boolean` — прокси к `API_CONFIGURED` из `./client`: есть ли вообще настроенный бэкенд (иначе `App.tsx` показывает демо-вход без сети).
- `primaryRole(roles): Role` — превращает список ролей бэкенда в одну роль интерфейса-прототипа через `ROLE_MAP` (admin/teacher/accountant/student), по умолчанию `'admin'`, если ни одна роль не распознана.
- `apiLogin(tenant, email, password): Promise<SessionUser>` — `POST /api/v1/auth/login`; при ошибке пробрасывает `ApiError` (401 — неверные данные, 429 — превышен лимит попыток).
- `apiMe(): Promise<SessionUser | null>` — `GET /api/v1/auth/me`; не бросает исключений — возвращает `null` при любой ошибке (включая отсутствие сессии), используется при старте приложения, чтобы восстановить сессию без явного логина.
- `apiLogout(): Promise<void>` — `POST /api/v1/auth/logout`; best-effort — гасит серверную сессию, но ошибки проглатывает, потому что logout считается идемпотентным.

## Как это работает

Все запросы идут через `apiFetch` из `./client`, который автоматически отправляет `credentials: 'include'`, поэтому клиенту не нужно ничего знать про сам токен сессии — только про её наличие (успех/ошибка запроса). `primaryRole` — мост между настоящей RBAC-моделью бэкенда (произвольный список строковых ролей) и упрощённой одной ролью, на которой построен весь UI-прототип NEX.

## Связи

Использует `apiFetch`, `API_CONFIGURED` из `./client` и тип `Role` из `../data`. Потребляется `App.tsx` (экран `Login`, кнопка «Выйти» в `MobileShell`) и реэкспортируется через `web/src/api/index.ts` как пространство имён `auth`.

## На что обратить внимание

`apiMe` и `apiLogout` при `!API_CONFIGURED` даже не пытаются сходить в сеть (`return null`/`return` сразу) — это защита демо-режима от лишних сетевых запросов на несуществующий бэкенд, а не просто оптимизация.
