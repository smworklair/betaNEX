# web/src/api/client.ts

Ядро всего API-слоя NEX: единая точка входа для HTTP-запросов к бэкенду `nexd` — базовый URL, куки-сессии, разбор ошибок в формате RFC 9457 (Problem Details) и механизм автоматического фолбэка на встроенные моки, когда бэкенд не настроен или недоступен. Все остальные файлы `web/src/api/*` строятся поверх этого файла.

## Ключевое

- `API_BASE` — базовый URL для запросов, вычисляется из `VITE_API_URL`; пустая строка означает «тот же origin».
- `API_CONFIGURED: boolean` — есть ли вообще настроенный бэкенд; если нет, весь слой `api/` работает на моках без единого сетевого запроса.
- `class ApiError extends Error` — несёт HTTP `status` и человекочитаемые `title`/`detail` из тела `application/problem+json`.
- `apiFetch<T>(path, init): Promise<T>` — низкоуровневый запрос: всегда шлёт куки (`credentials: 'include'`), проставляет `Content-Type: application/json` при наличии тела, разбирает JSON-ответ, при не-2xx бросает `ApiError`; `204 No Content` превращается в `undefined`.
- `withFallback<T>(call, mock): Promise<T>` — обёртка для операций ЧТЕНИЯ: при `!API_CONFIGURED` сразу возвращает мок, при ошибке реального вызова (сеть, 5xx, нет сессии) — тоже тихо возвращает мок (с предупреждением в консоли, если `VITE_API_URL` был задан).
- `apiStatus(): { configured, base }` — сводка о состоянии подключения для диагностики в UI.

## Как это работает

`apiFetch` — единственное место, где происходит реальный `fetch` к серверу; всё остальное (`auth.ts`, `campus.ts`, `tasks.ts`, `terminal.ts`) вызывает только его. `withFallback` реализует главный принцип UX прототипа: экраны, читающие данные, никогда не остаются пустыми и не падают — если бэкенда нет или он недоступен, они прозрачно получают встроенные демо-данные. Явно оговорено, что мутации (login, create, complete, delete) этот фолбэк НЕ используют — их результат обязан отражать настоящий ответ сервера, иначе пользователь решит, что действие выполнилось, хотя оно ушло в никуда.

## Связи

Не имеет зависимостей внутри проекта (только Vite `import.meta.env`). От него зависят буквально все остальные файлы `web/src/api/*` (`auth.ts`, `campus.ts`, `tasks.ts`, `terminal.ts`, `agents.ts` косвенно) и реэкспортирующий их `index.ts`.

## На что обратить внимание

Различие между `apiFetch` (голый вызов, всегда пробрасывает ошибку) и `withFallback` (только для чтений, глотает ошибку в пользу мока) — ключевой архитектурный инвариант этого слоя: если в новом коде понадобится обёртка над мутацией, использовать `withFallback` для неё нельзя — так теряется единственный сигнал пользователю о реальном сбое операции.
