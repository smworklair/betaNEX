# web/src/api/index.ts

Единая точка входа в API-слой NEX: barrel-файл, через который весь остальной фронтенд импортирует доступ к бэкенду — `import { tasksApi, campusApi, auth, apiStatus } from '../api'` — вместо того, чтобы тянуть каждый модуль (`./auth`, `./campus`, `./tasks`) напрямую по отдельности.

## Ключевое

- Реэкспорт инфраструктуры напрямую: `API_BASE`, `API_CONFIGURED`, `ApiError`, `apiFetch`, `withFallback`, `apiStatus` из `./client`.
- `export * as auth from './auth'` — модуль аутентификации как пространство имён.
- `export * as tasksApi from './tasks'` — модуль задач как пространство имён.
- `export * as campusApi from './campus'` — модуль кампуса как пространство имён.
- `export * as agentsApi from './agents'` — модуль агентов как пространство имён.

## Как это работает

Доменные модули реэкспортируются через `export * as <name>`, а не поимённо, потому что у разных модулей есть одноимённые функции (например, `listTasks`/`listStudents`/`listAgents` могли бы называться просто `list`, но и так возможны коллизии по смыслу) — обёртка в пространство имён (`tasksApi.listTasks()`, `campusApi.listStudents()`) убирает риск конфликта имён при плоском реэкспорте.

## Связи

Просто агрегирует `./client`, `./auth`, `./tasks`, `./campus`, `./agents` — сам не содержит логики. Импортируется практически всеми компонентами фронтенда, которым нужны данные с бэкенда (`App.tsx` использует `auth`, страницы разделов используют `campusApi`/`tasksApi`/`agentsApi`).

## На что обратить внимание

Файл нужно обновлять при добавлении каждого нового доменного модуля API (например, будущего `terminal.ts` тут пока нет в реэкспорте — он импортируется напрямую как `./terminal` там, где нужен) — не все файлы `api/` обязательно проходят через этот barrel.
