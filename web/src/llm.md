# web/src/llm.ts

Единая точка входа к ИИ для фронтенда NEX — все вызовы уходят в отдельный бэкенд-сервис `ai-gateway/` (Python, см. `ai-gateway/README.md`), а не напрямую во внешние LLM-провайдеры. Ключи провайдеров (Gemini/OpenAI/DeepSeek/...) живут только в переменных окружения `ai-gateway` — во фронтенде их больше нет вообще. Конфигурация — переменная `VITE_AI_GATEWAY_URL` (см. `.env.example`): пусто — демо-режим без сети, URL шлюза — реальные вызовы.

## Ключевое

- `AI_GATEWAY_BASE` / `AI_GATEWAY_CONFIGURED` — базовый URL шлюза и признак, что он сконфигурирован.
- `LlmProvider` — тип имени провайдера на шлюзе (`'gemini' | 'custom' | 'openai' | 'deepseek' | 'qwen' | 'kimi' | 'gigachat' | 'yandexgpt'`).
- `getProvider`/`setProvider` — только ПРЕДПОЧТЕНИЕ провайдера (имя, без ключей) в localStorage; пусто — шлюз берёт провайдера по умолчанию из своего конфига.
- `llmReady()` — готов ли ИИ отвечать, то есть сконфигурирован ли шлюз (ключи больше не проверяются — их просто нет на клиенте).
- `ORG_CONTEXT` — большой системный промпт главного чата NEX (`Chat.tsx`): личность, режимы ответа, демо-данные организации. Передаётся как явный `system`-оверрайд — в отличие от мини-чатов на страницах (`AiBox`/`InlinePanel`), которые вместо этого шлют `context` и получают промпт раздела с сервера.
- `PageContext` — контекст страницы `{ page, title?, facts?, state? }`: `page` превращается на сервере в ролевую инструкцию (см. `ai-gateway/app/core/context_registry.py`), `facts`/`state` — живые данные экрана.
- `llmAsk(user, opts)` — единая точка входа: `POST {AI_GATEWAY_BASE}/api/v1/ai/ask` с `{message, history, system?, context?, provider?}`, бросает исключение при ошибке/не-конфигурации.
- `fetchProviders()` — список провайдеров, реально настроенных на сервере (`GET /api/v1/ai/providers`), для выбора в Настройках.
- `checkGateway()` — жив ли шлюз (`GET /healthz`) — для индикатора статуса в Настройках.
- `llmPlan(task)` — просит модель составить план из 3-5 шагов для задачи («Поручить NEX»), парсит ответ построчно.

## Как это работает

`llmAsk` собирает тело запроса (историю переводит в формат `{role: 'user'|'assistant', content}`, добавляет `context`/`system`/`provider`, если заданы) и шлёт один `fetch` в ai-gateway с таймаутом через `withTimeout` (AbortController). При ошибке (шлюз не сконфигурирован, не-2xx ответ, пустой текст) выбрасывается исключение — вызывающий код обязан откатиться на локальный мок-слой (`nexbrain.ts`), этот файл фолбэком сам не занимается.

## Связи

Не зависит от других файлов проекта (только `fetch`/localStorage/`import.meta.env`). Используется компонентами, которые обращаются к ИИ-помощнику NEX: `web/src/nexbrain.ts` (локальный фолбэк), `web/src/pages/Settings.tsx` (выбор провайдера из списка сервера + статус шлюза), `web/src/pages/aibox.tsx` и `web/src/ai.tsx` (мини-чаты с контекстом раздела и своей историей), `web/src/pages/Chat.tsx`, `web/src/terminal.tsx`, `web/src/pages/Home.tsx`.

## На что обратить внимание

`ORG_CONTEXT` по-прежнему содержит захардкоженные демо-данные организации — заглушка для прототипа, в реальной системе должна замениться на RAG по настоящей БД (см. `docs/ai/README.md`, §2.6). Подробный разбор архитектуры (что сегодня, что в планах) — там же, `docs/ai/README.md`.
