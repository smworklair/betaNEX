# web/src/llm.ts

Слой интеграции с внешними LLM-провайдерами для ИИ-помощника NEX. Поддерживает два варианта на выбор пользователя (настраивается в Настройки → Интеллект): Google Gemini (`gemini-2.5-flash`) и произвольный OpenAI-совместимый endpoint (`/chat/completions`). Ключи API вводятся пользователем в настройках и хранятся только в localStorage браузера — не попадают ни в репозиторий, ни на сервер. Комментарий в начале файла прямо говорит, что в проде этот файл заменится на вызов `/api/nex`, где ключ будет лежать на backend в переменных окружения, а не у клиента — то есть сейчас это временное клиентское решение ради прототипа.

## Ключевое

- `LlmProvider` — тип `'gemini' | 'custom'`.
- `getProvider`/`setProvider` — какой провайдер активен сейчас.
- `getGeminiKey`/`setGeminiKey` — ключ Gemini.
- `getCustomKey`/`setCustomKey`, `getCustomUrl`/`setCustomUrl`, `getCustomModel`/`setCustomModel` — настройки кастомного OpenAI-совместимого провайдера (по умолчанию `CUSTOM_DEFAULT_URL = 'https://llm-api.fun/v1'`, модель `agent`).
- `llmReady()` — готов ли активный провайдер отвечать (есть ли достаточно длинный ключ).
- `ORG_CONTEXT` — большой системный промпт: описывает личность NEX, режимы ответа (общение/аналитика/консультация/общие вопросы), стиль и демо-данные организации (студенты, финансы, безопасность и т.д.), которые модель использует как контекст по умолчанию.
- `llmAsk(user, opts)` — единая точка входа: отправляет вопрос активному провайдеру (Gemini или custom), бросает исключение при ошибке.
- `testLlmKey(provider, key, url?, model?)` — проверка валидности ключа коротким тестовым запросом (используется в настройках при вводе ключа).
- `llmPlan(task)` — просит модель составить план из 3-5 шагов для задачи (используется функцией «Поручить NEX»), парсит ответ построчно.

## Как это работает

`askGemini` и `askCustom` — приватные функции, реализующие конкретные HTTP-запросы к соответствующим API (Gemini `generateContent` и OpenAI-подобный `/chat/completions`), обе обёрнуты таймаутом через `withTimeout` (AbortController). `llmAsk` выбирает нужный путь через `getProvider()`. При ошибке (нет ключа, не-200 ответ, пустой текст) выбрасывается исключение — согласно комментарию к `llmAsk`, вызывающий код обязан откатиться на локальный мок-слой (`nexbrain.ts`), то есть этот файл не отвечает за фолбэк сам.

## Связи

Не зависит от других файлов проекта (только от `fetch` и localStorage). Используется компонентами, которые обращаются к ИИ-помощнику NEX — в частности `web/src/nexbrain.ts` (локальный фолбэк при недоступности LLM), экраном настроек `web/src/pages/Settings.tsx` (ввод и проверка ключей через `testLlmKey`), а также любыми местами с чатом/терминалом NEX (`web/src/pages/Chat.tsx`, `web/src/terminal.tsx`, `web/src/pages/Home.tsx`).

## На что обратить внимание

`ORG_CONTEXT` содержит захардкоженные демо-данные организации (конкретные студенты, суммы задолженностей, IP-адреса) — это заглушка для прототипа; в реальной системе описана как то, что должно замениться на RAG по настоящей БД.
