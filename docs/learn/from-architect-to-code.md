# От архитектора к коду: заполняем недостающий слой

Этот документ — для конкретного профиля: ты уже мыслишь системами —
границы, контракты, потоки данных, транзакционность, трейд-оффы между
консистентностью и доступностью. Эта часть пирамиды у тебя есть и
хорошо построена. Но синтаксис, тулинг и механика письма кода — та
часть, которую обычно нарабатывают годами практики, — пока пустая
полка. Этот документ её заполняет, и делает это **не заново объясняя
архитектуру**, а показывая, как то, что ты уже понимаешь, буквально
записывается символами на экране.

Формула документа: **ты уже знаешь ЧТО — здесь только КАК записать.**
Если в тексте попадётся объяснение архитектурной концепции — это
ошибка документа, пишите issue; здесь должна быть только нотация.

Место в учебном цикле: читай это **параллельно** с
[`how-to-read-and-learn.md`](how-to-read-and-learn.md), держа открытым
как словарь, когда там написано «откройте файл X» — сначала находишь
файл в таблице раздела 1 (какую архитектурную роль он играет — это ты
и так поймёшь по подсказке), потом при необходимости лезешь в раздел
2-3 за расшифровкой конкретного синтаксиса, который в нём встретится.

## 1. Таблица-переводчик: концепт → место в NEX → как это записано

### Границы и структура

| Архитектурный концепт (ты его знаешь) | Где в NEX | Как это буквально записано |
|---|---|---|
| Bounded context / граница модуля | `internal/module/finance/`, `internal/module/tasks/` | Каталог + строка `package finance` в начале каждого файла внутри него. Модуль в Go — это **просто каталог с общим `package`**, никакой отдельной сущности «модуль» в языке нет. |
| Слоистая архитектура (domain/application/infra) | `internal/kernel` / `internal/module` / `internal/platform` | Три каталога. Направление зависимостей — НЕ обеспечено языком автоматически, это соглашение, которое держится код-ревью (и потенциально линтером `depguard`, см. `../go-guide.md` §29). Кто на самом деле от кого зависит — видно по секции `import (...)` в начале файла: если `internal/module/finance/handlers.go` импортирует `internal/kernel/command`, значит finance зависит от kernel. |
| Композиционный корень (где всё связывается) | `cmd/nexd/main.go` | Обычная функция `func serve(...)`, которая по порядку **создаёт значения и передаёт их друг другу как аргументы**. Никакого «DI-контейнера» как отдельного механизма — просто явные вызовы функций, результат одной идёт в параметр другой. |
| Модуль/пакет как единица переиспользования (фронтенд) | `web/src/api/`, `web/src/features/ai/` | Каталог + файлы с `export` — то, что экспортировано, видно другим файлам через `import { x } from '../api'`. |

### Контракты и реализации

| Концепт | Где в NEX | Синтаксис |
|---|---|---|
| Контракт / порт (hexagonal architecture) | `internal/kernel/command/command.go` — интерфейс `Command` | `type Command interface { Name() string; Permission() string; Validate() error }` — просто список сигнатур методов без тела. Это ВЕСЬ контракт. |
| Адаптер, реализующий контракт | `internal/module/finance/pgrepo.go: PostgresRepository` реализует `finance.Repository` | В Go **нет ключевого слова `implements`**. Если у структуры есть методы с ровно теми же сигнатурами, что в интерфейсе, — она ему **автоматически** соответствует (structural typing / duck typing на уровне компилятора). Компилятор сам проверяет это в месте использования, а не в месте объявления структуры. |
| Порт, который реализует ИМЕННО потребитель (Dependency Inversion) | `command.TxRunner` объявлен в `kernel/command`, реализован в `platform/postgres` | Тот же приём: интерфейс лежит в пакете, который его ИСПОЛЬЗУЕТ, а не в пакете, который его реализует — так пакет-потребитель не импортирует пакет-реализацию (`docs/go-guide.md` §1, «Интерфейсы объявляет потребитель»). |
| Value object / неизменяемая структура данных | `internal/kernel/audit` — запись аудита | `type` + `struct { Поле Тип; ... }` — просто именованная группа полей. «Неизменяемость» в Go не отдельное ключевое слово, а соглашение: если функция принимает структуру по значению (не по указателю), у неё своя копия. |

### Управление потоком и связыванием

| Концепт | Где в NEX | Синтаксис |
|---|---|---|
| Message bus / шина команд | `internal/kernel/command/bus.go` | Структура `MemoryBus` хранит `map[string]HandlerFunc` (обычный словарь «имя команды → функция-обработчик»). `Dispatch` — метод, который достаёт функцию из словаря по ключу и **вызывает её как обычную функцию**: `handler(ctx, cmd)`. Функции в Go — «граждане первого класса»: их можно класть в переменные/поля структур и вызывать через них. |
| Транзакционная граница (unit of work) | `internal/platform/postgres/tx.go: InTenantTx`, `command.TxRunner.RunTx` | Функция, принимающая **другую функцию** как аргумент: `RunTx(ctx, func(ctx context.Context) error { ... })`. Это называется функция высшего порядка (higher-order function) — паттерн «передай мне то, что нужно выполнить внутри транзакции, я оберну это BEGIN/COMMIT/ROLLBACK». |
| Retry / circuit breaker | `ai-gateway/app/core/retry.py` | Python: обычная функция с циклом `for attempt in range(...)`, `await asyncio.sleep(delay)` между попытками, `try/except` вокруг вызова. Ничего специального в языке — просто цикл и обработка исключений. |
| Rate limiting (алгоритм fixed-window) | `internal/platform/httpapi/ratelimit.go` | Структура со `map[string]int` (счётчик по ключу) + `sync.Mutex` (замок на структуру данных, чтобы две горутины не испортили счётчик одновременно). |
| Идемпотентность (повтор не создаёт дубль) | `internal/platform/httpapi/idempotency.go` | Таблица в БД `(ключ → сохранённый ответ)` + проверка «есть ли уже такой ключ» ДО выполнения операции. Архитектурно — кэш с ключом от клиента, а не от сервера. |

### Состояние и данные (фронтенд)

| Концепт | Где в NEX | Синтаксис |
|---|---|---|
| Глобальный контейнер состояния (state container) | `web/src/ui.tsx: AppProvider`/`useApp()` | React `createContext()` создаёт «трубу», в которую `AppProvider` кладёт значения через `useState` (переменная + функция для её изменения), а любой дочерний компонент читает через `useContext()` (обёрнуто в `useApp()`). |
| Реактивность / автоматический перерендер при изменении данных | Любой компонент с `useState` | Вызов функции-сеттера (`setPage(id)`) не просто меняет переменную — он **планирует повторный вызов** функции-компонента React'ом. Не нужно вручную перерисовывать DOM. |
| Побочный эффект при изменении зависимости | `App.tsx`, `ui.tsx` — `useEffect(() => {...}, [зависимости])` | Функция первым аргументом выполняется после рендера; массив вторым аргументом — «выполнить заново, если что-то из этого списка изменилось со времени прошлого рендера». |
| Ленивая загрузка / асинхронная граница модуля | `App.tsx: PAGES`, `React.lazy(() => import('./pages/...'))` | `import(...)` как **функция**, а не как ключевое слово вверху файла — возвращает `Promise` с модулем, подгружаемым по сети при первом вызове. |

### Конкурентность и асинхронность

| Концепт | Где в NEX | Синтаксис |
|---|---|---|
| Единица конкурентного выполнения (легковесный поток) | `cmd/nexd/main.go: runServers` (HTTP-сервер, cron, outbox — параллельно) | `go func() { ... }()` — запускает функцию, не дожидаясь её завершения, дальнейший код выполняется сразу. Это и есть горутина. |
| Ожидание завершения нескольких параллельных задач + первая ошибка отменяет остальные | `errgroup.WithContext` в `runServers` | Обычная библиотека поверх горутин + каналов: `g.Go(func() error {...})`, `g.Wait()` возвращает первую ошибку. |
| Взаимное исключение (защита общих данных) | `internal/kernel/command/bus.go: MemoryBus.handlers`, `ratelimit.go` | `sync.Mutex` + `mu.Lock()`/`defer mu.Unlock()` вокруг чтения/записи общей структуры. |
| Асинхронный конвейер / поток данных (streaming) | `web/src/features/ai/llm.ts: llmAskStream` | `async function`, `await reader.read()` в цикле, обработка `done`/`value` — JS-эквивалент чтения из канала порциями. |

### Конфигурация и окружение

| Концепт | Где в NEX | Синтаксис |
|---|---|---|
| 12-factor конфигурация из окружения | `internal/config/config.go` | `os.LookupEnv("NEX_HTTP_ADDR")` — читает переменную окружения процесса как строку, дальше вручную парсится в нужный тип (`time.ParseDuration` для таймаутов и т.п.). |
| Feature flag | `VITE_AI_ENABLED` (фронтенд), `NEX_AI_GATEWAY_URL` (бэкенд, пусто = прокси не монтируется) | Просто `if` по значению переменной окружения/конфига — никакого специального механизма флагов в этом проекте нет. |
| RBAC-политика (роль → право) | `internal/kernel/authz` | `map[string]map[string]bool` по сути (реализовано через структуру `Policy`) — обычный словарь словарей. |

## 2. Недостающий джуниорский слой — быстрое заполнение по языкам

Каждый пункт — то, что джун знает не задумываясь, а тебе нужно
установить как факт. Без разжёвывания «зачем» — оно у тебя уже есть из
раздела 1, здесь только «как это выглядит».

### 2.1 Go — минимум синтаксиса

```go
// Переменные: явный тип ИЛИ короткое объявление с выводом типа
var addr string = ":8080"
addr := ":8080"          // то же самое, тип выводится из значения

// Функция: func имя(параметры) (возвращаемые типы) { тело }
func Connect(ctx context.Context, dsn string) (*DB, error) {
    // ...
    return &DB{pool: pool}, nil   // nil = "ошибки нет", успешный путь
}

// Структура — просто именованная группа полей
type Actor struct {
    ID    string
    Roles []string   // []T = слайс (динамический массив) элементов типа T
}

// Метод — функция, "привязанная" к типу через приёмник в скобках перед именем
func (a Actor) LogValue() slog.Value { /* ... */ }
//    ^ receiver: "a" играет роль self/this внутри метода

// Интерфейс — список сигнатур методов, БЕЗ реализации
type Repository interface {
    Accounts(ctx context.Context) ([]Account, error)
}
// Любая структура с методом Accounts(ctx) ([]Account, error) АВТОМАТИЧЕСКИ
// подходит под этот интерфейс — ключевое слово "implements" не нужно и не существует.

// Ошибки — обычные значения, а не исключения
res, err := someCall()
if err != nil {
    return fmt.Errorf("контекст: %w", err)   // %w оборачивает исходную ошибку
}
// errors.Is(err, ErrNotFound) — проверка "это та самая ошибка (или обёртка над ней)?"

// Слайсы и мапы — динамический массив и словарь
items := []string{"a", "b"}
byID := map[string]int{"a": 1}

// for — единственная конструкция цикла в Go (нет while/do-while)
for i, item := range items { /* i — индекс, item — значение */ }

// Указатель — "адрес переменной", нужен, чтобы функция могла делиться
// одним и тем же объектом (например, пулом соединений к БД), а не копировать его
func (d *DB) Ready(ctx context.Context) error { /* d — общий на всех вызовах */ }

// Горутина — запустить функцию не дожидаясь её завершения
go func() { sched.Run(ctx) }()
```

### 2.2 TypeScript / React — минимум синтаксиса

```ts

const API_BASE = '/api';
let count = 0;

function add(a: number, b: number): number { return a + b; }
```

### 2.3 Python — минимум синтаксиса (для `ai-gateway/`, `beta/`)

```python
# Отступы вместо {} — блок кода определяется отступом, это не стиль, а синтаксис
def complete(self, messages, system=None):
    if not messages:
        raise ValueError("messages required")
    return result

# self — явный первый параметр метода (аналог "приёмника" в Go/this в JS)
class OpenAICompatProvider(LLMProvider):
```

### 2.4 Как читать сообщение об ошибке / стектрейс

Это отдельный навык, который джуниор нарабатывает интуитивно, а тебе
нужно освоить сознательно: **сообщение об ошибке почти всегда точно
называет файл и строку — задача не «понять код», а найти нужную строку
и прочитать одно предложение рядом с ней.**

- **Go, ошибка компиляции** — формат `путь/к/файлу.go:строка:колонка:
  сообщение`, например `internal/module/tasks/http.go:42:10: undefined:
  taskRepo`. Открывай файл, переходи на указанную строку — компилятор
  не ошибается в месте, ошибается почти всегда в диагнозе (что именно
  не так может быть неточно описано, но где — всегда точно).
- **Go, паника в рантайме (stack trace)** — блок вида
  `goroutine 1 [running]: package.Func(...) /путь/файл.go:строка`.
  Читай **сверху вниз**: самая верхняя строка — где паника случилась
  фактически; ищи первую строку, указывающую на твой код (каталоги
  `internal/`, `cmd/`), а не на код стандартной библиотеки — обычно
  фикс нужен именно там.
- **Браузер (TS/JS в рантайме)** — вкладка Console в DevTools; красный
  текст, обычно с `at имяФункции (файл:строка:колонка)` — кликабельная
  ссылка сразу открывает исходник в этой же строке. Вкладка Network
  рядом полезна отдельно: там видно, ушёл ли вообще запрос к
  `/api/v1/...` и какой пришёл статус.
- **Python-traceback** — читай **снизу вверх**: последняя строка —
  тип и текст самой ошибки (`ValueError: ...`), строки выше — цепочка
  вызовов, приведшая к ней, самая нижняя из них (ближе к концу) —
  обычно твой код, а не библиотека.

### 2.5 Пакетные менеджеры — три штуки, один и тот же смысл

Во всех трёх случаях идея одна: **файл-манифест** объявляет, какие
внешние зависимости нужны проекту, **lock-файл** фиксирует точные
версии для воспроизводимости у всех разработчиков, команда
устанавливает то, что описано в манифесте.

| | Манифест | Lock-файл | Установить | Добавить зависимость | Запустить |
|---|---|---|---|---|---|
| **Go** (`go.mod`, корень) | `go.mod` (имя модуля, версия Go, прямые зависимости) | `go.sum` (хэши для проверки целостности) | не нужна отдельная команда — `go build`/`go run` сами скачивают недостающее | `go get пакет`, затем `go mod tidy` (убирает лишнее/подтягивает нужное) | `go run ./cmd/nexd` — компилирует и сразу выполняет, без отдельного шага сборки |
| **pnpm** (`web/package.json`) | `package.json` (зависимости + `scripts`, например `"dev": "vite"`) | `pnpm-lock.yaml` (корень репозитория — монорепо, `pnpm-workspace.yaml`) | `pnpm install` | `pnpm add пакет` | `pnpm dev` (или из корня: `pnpm -C web dev` — выполнить скрипт `dev` внутри каталога `web`) |
| **pip** (`ai-gateway/requirements.txt`, `beta/*/requirements.txt`) | `requirements.txt` (список пакетов с версиями) | обычно отдельного lock-файла нет — версии фиксированы прямо в `requirements.txt` | `pip install -r requirements.txt` | дописать строку в `requirements.txt`, переустановить | `python -m app.main` / `uvicorn app.main:app` |

**Зачем `venv` в Python, если в Go/Node такого нет?** Python по
умолчанию ставит пакеты в одно общее место на систему — два проекта с
разными версиями одной библиотеки конфликтуют. `venv` создаёт
изолированную копию интерпретатора+пакетов на один проект (папка
`.venv/`, активируется `source .venv/bin/activate` или
`.venv\Scripts\Activate.ps1` на Windows). Go и Node решают ту же
проблему иначе (модульная система с версиями внутри проекта), поэтому
отдельного шага «активации окружения» им не нужно.

### 2.6 Переменные окружения

Переменная окружения — пара «имя=значение», которую видит только что
запущенный процесс (и его дочерние процессы), а не файл на диске.
Задать её можно тремя способами, все три используются в этом
репозитории:

```sh
# Bash: одноразово перед командой
NEX_LOG_LEVEL=debug go run ./cmd/nexd

# PowerShell: на всю текущую сессию терминала
$env:NEX_LOG_LEVEL = "debug"

# Файл .env + docker-compose.yaml: значения из .env подставляются
# автоматически в блок environment: (см. compose.yaml)
```

В коде это читается предельно просто — никакой магии:
`internal/config/config.go` вызывает `os.LookupEnv("NEX_HTTP_ADDR")` и
получает строку (или пустую строку, если не задано — отсюда идея
«разумных дефолтов» для каждой переменной). `.env.example` в каждом
каталоге (корень, `web/`, `ai-gateway/`) — это просто шаблон, который
копируют в `.env` (`cp .env.example .env`) и правят; сам `.env` — в
`.gitignore`, потому что в нём могут оказаться секреты.

### 2.7 Git — минимальный набор команд

Репозиторий — это папка, у которой есть **история снимков** (коммитов).
Минимум, которого достаточно для повседневной работы:

```sh
git status              # что изменено с последнего снимка
git diff                # что именно изменилось, построчно
git add путь/к/файлу     # подготовить файл к следующему снимку
git commit -m "текст"    # сделать снимок с сообщением
git log --oneline        # история снимков, коротко
git branch               # список веток (независимых линий истории)
git checkout -b моя-фича  # создать и переключиться на новую ветку
git pull                 # подтянуть чужие снимки с сервера
git push                 # отправить свои снимки на сервер
```

Ветка (`branch`) — это просто именованный указатель на конкретный
коммит, который двигается вперёд с каждым новым коммитом в ней;
переключение между ветками (`git checkout`) меняет содержимое файлов
на диске на то, что было зафиксировано в этой ветке. В этом проекте
изменения проходят через PR с обязательным зелёным CI
(`../../CONTRIBUTING.md`) — на практике это и означает работу в
отдельной ветке, а не прямые коммиты в `main`.

### 2.8 Как вообще запустить и отладить

Практический цикл, который замыкает всё вышесказанное — подробные
команды запуска каждого сервиса уже собраны в
[`getting-started.md`](getting-started.md), здесь — сама механика
процесса «запустил → не работает → почему»:

1. **Запуск — это просто выполнение команды в терминале**, не какое-то
   отдельное «действие IDE»: `make run` (Go), `pnpm dev` (фронтенд),
   `uvicorn app.main:app --reload` (Python) — все они читают
   исходный код заново, компилируют/интерпретируют и начинают слушать
   порт. Ctrl+C в терминале — остановить процесс.
2. **Самый дешёвый отладчик — вывод текста.** Не обязательно сразу
   ставить точки останова (breakpoints) в IDE: `fmt.Println("здесь:",
   значение)` в Go, `console.log(значение)` в TS, `print(значение)` в
   Python — добавь временную строку рядом с местом, которое не
   понимаешь, перезапусти, посмотри вывод в терминале/консоли браузера,
   удали строку, когда разобрался. Джуниоры называют это «принт-дебаг»
   и пользуются им постоянно — это не «неправильный» способ отладки.
3. **Проверить, что вообще компилируется, не запуская** — `go build
   ./...` (Go) и `pnpm lint` (это `tsc --noEmit` — проверка типов без
   сборки, см. `web/package.json`) — быстрее, чем полный перезапуск,
   если ищешь именно синтаксическую/типовую ошибку.
4. **Проверить бэкенд напрямую, без браузера** — `curl` бьёт по
   конкретному эндпоинту и показывает ровно то, что вернул сервер,
   без слоя UI между тобой и ответом:
   ```sh
   curl -s http://localhost:8080/healthz
   curl -s -X POST http://localhost:8080/api/v1/auth/login \
     -H "Content-Type: application/json" \
     -d '{"tenant":"college-1","email":"admin@example.ru","password":"..."}' -i
   ```
5. **Логи сервиса — первое место смотреть, если что-то «просто не
   работает»** без явной ошибки в терминале — `nexd` и `ai-gateway`
   пишут структурные JSON/текстовые логи с `request_id` (см.
   `../observability.md`); в Docker — `docker compose logs -f nexd`.

## 3. Разобрано построчно: один реальный кусок кода целиком

Лучший способ закрепить раздел 2 — разобрать не выдуманный пример, а
настоящий фрагмент из репозитория, который ты уже видел как
«архитектурный пример» в `backend-go.md`. Это главный доменный инвариант
finance-модуля, `internal/module/finance/commands.go`:

```go
func (c PostEntry) Validate() error {
    if len(c.Lines) < 2 {
        return errors.New("finance: entry needs at least two lines")
    }
    var debit, credit int64
    for i, l := range c.Lines {
        if l.Amount <= 0 {
            return fmt.Errorf("finance: line %d: amount must be positive", i)
        }
        switch l.Side {
        case Debit:
            debit += l.Amount
        case Credit:
            credit += l.Amount
        default:
            return fmt.Errorf("finance: line %d: unknown side %q", i, l.Side)
        }
    }
    if debit != credit {
        return fmt.Errorf("%w: debit %d != credit %d", ErrUnbalanced, debit, credit)
    }
    return nil
}
```

Построчно (это ровно та реализация контракта `Command.Validate() error`
из раздела 1):

- `func (c PostEntry) Validate() error` — метод на типе `PostEntry`;
  `c` — «this»; возвращает одно значение типа `error` (интерфейс из
  одного метода `Error() string`, тебе не нужно писать его самому —
  `errors.New`/`fmt.Errorf` уже создают такие значения).
- `errors.New("...")` — создать ошибку-значение с текстом. Ничего не
  «бросается» — это просто конструктор значения, как `Account{...}`.
- `var debit, credit int64` — две переменные явного типа «64-битное
  целое» — деньги в NEX всегда целые (минорные единицы, копейки), не
  `float`, это архитектурное решение из `../decision-log.md` (ADR),
  записанное здесь буквально типом переменной.
- `for i, l := range c.Lines` — цикл по слайсу `c.Lines`: `i` — индекс
  (0, 1, 2...), `l` — копия элемента на этой итерации.
- `switch l.Side { case Debit: ...; case Credit: ...; default: ... }`
  — сопоставление значения с образцом; `default` — «во всех остальных
  случаях» (здесь это защита от несуществующего значения `Side`).
- `fmt.Errorf("%w: debit %d != credit %d", ErrUnbalanced, debit, credit)`
  — создаёт новую ошибку, которая ОБОРАЧИВАЕТ `ErrUnbalanced` (`%w`) —
  вызывающий код может проверить `errors.Is(err, ErrUnbalanced)` и
  узнать «это была именно ошибка баланса», не разбирая текст строкой.
  Это Go-эквивалент иерархии классов исключений в других языках —
  здесь роль «типа ошибки» играет обёрнутое значение, а не класс.
- `return nil` — `nil` для интерфейса `error` означает «ошибки нет,
  всё в порядке» — это единственный «успешный» путь функции.

И для контраста — TS-фрагмент из `web/src/api/client.ts`
(`apiFetch`), тот же приём разбора:

```ts
export async function apiFetch<T>(path: string, init: RequestInit = {}): Promise<T> {
  const res = await fetch(API_BASE + path, { credentials: 'include', ...init });
  if (res.status === 204) return undefined as T;
  // ...
}
```

- `export` — эта функция видна другим файлам через `import`, это
  публичная граница модуля `api/client.ts` (см. раздел 1 — «модуль как
  единица переиспользования»).
- `async function ... : Promise<T>` — функция асинхронная; возвращает
  не сам результат немедленно, а «обещание» результата типа `T`,
  которое разрешится позже.
- `<T>` — generic-параметр: эта функция работает для ЛЮБОГО типа
  ответа `T`, конкретный тип подставляет вызывающий код
  (`apiFetch<Task[]>(...)`).
- `await fetch(...)` — поставить выполнение на паузу здесь, пока сеть
  не ответит, затем продолжить со значением ответа.
- `{ credentials: 'include', ...init }` — объектный литерал;
  `...init` — spread-оператор, «разложи все поля объекта `init` сюда
  же» (аналог слияния двух словарей).
- `undefined as T` — приказ компилятору «доверься мне, здесь тип `T»`
  (проверяется только на этапе компиляции, в рантайме ничего не
  проверяет).

## 4. ИИ как переводчик в обе стороны

Твой главный сценарий использования ИИ отличается от «обычного»
онбординга: тебе не нужно объяснение архитектуры (ты её уже
понимаешь) — тебе нужен **переводчик между двумя нотациями**: та, что
у тебя в голове (границы, контракты, потоки), и та, что на экране
(ключевые слова, скобки, отступы).

### 4.1 Направление «концепт → синтаксис»

Формулируй запрос от архитектурной идеи, явно говоря, что понимаешь
«зачем», и прося только «как»:

```
Мне нужен ретрай с экспоненциальной задержкой перед вызовом внешнего
HTTP API в Go — я понимаю, зачем он нужен (транзиентные сбои сети),
не знаю точного синтаксиса. Дай идиоматичный код на stdlib (в проекте
принцип stdlib-first, см. docs/go-guide.md §5 — не предлагай внешнюю
библиотеку). Покажи, как оформить это как функцию, которую можно
переиспользовать, а не как ad-hoc цикл в одном месте.
```

```
Я хочу объявить контракт (интерфейс) для отправки уведомлений — один
метод Send(). Дай точный Go-синтаксис: объявление интерфейса, структуру
конкретной реализации (например, EmailSender), и как подключить это в
cmd/nexd/main.go по тому же образцу, что там уже подключён
finance.Repository (посмотри buildMounts/setupInfra перед ответом).
```

### 4.2 Направление «синтаксис → концепт»

Формулируй так, чтобы ИИ не объяснял базовые термины, а сразу называл
архитектурную роль конкретной строки — это и есть перевод в обратную
сторону:

```
Прочитай internal/module/finance/pgrepo.go целиком. Не объясняй мне,
что такое интерфейс, репозиторий или транзакция — я знаю эти концепты.
Вместо этого построчно скажи: вот эта конкретная строка — это
реализация какого архитектурного контракта, а вот эта — просто
техническая деталь синтаксиса Go (обработка ошибок SQL и т.п.), не
несущая архитектурного смысла.
```

```
Прочитай web/src/features/ai/llm.ts:llmAskStream. Я понимаю, что это
асинхронный конвейер потоковой обработки с ручной сборкой буфера. Покажи
мне, В КАКИХ конкретно строках JS/TS выражены: (1) чтение чанка из
потока, (2) накопление буфера до полного SSE-фрейма, (3) обработка
обрыва соединения. Мне нужна привязка "концепт → строка", не описание
конвейера в целом.
```

### 4.3 Отладочный цикл (когда что-то не работает)

Это самый частый практический сценарий на первых неделях — используй
ИИ, чтобы превратить непонятное сообщение об ошибке в понятную
архитектурную локацию, ПЕРЕД тем как просить исправление:

```
Вот текст ошибки после `go run ./cmd/nexd`:
<вставить сообщение целиком, включая путь к файлу и номер строки>

Прежде чем предлагать исправление, скажи: в каком слое это произошло
(kernel/module/platform/cmd — см. docs/learn/backend-go.md §1) и что
конкретно означает эта ошибка человеческим языком. Потом — минимальный
диф, а не переписанный файл целиком.
```

```
Браузер показывает в консоли: <вставить текст ошибки из DevTools>.
Это ошибка на этапе компиляции TypeScript, в рантайме React, или сетевой
сбой запроса к nexd? Как отличить эти три случая по виду сообщения —
объясни один раз, чтобы дальше я мог сам классифицировать похожие ошибки.
```

### 4.4 Как проверять ответы — с поправкой на твой профиль

Общий протокол проверки — в [`how-to-read-and-learn.md`](how-to-read-and-learn.md),
§4.4. Но у тебя есть один недостающий инструмент, который есть у
джуниора: **глаз, различающий правильный и неправильный синтаксис на
вид.** Пока его нет — не полагайся на «это выглядит разумно», полагайся
на внешних судей, которые не умеют льстить:

1. **Компилятор/тайпчекер — твой первый и самый дешёвый рецензент.**
   `go build ./...` (или `go vet ./...`) для Go, `pnpm lint` (это
   `tsc --noEmit`, см. `web/package.json`) для фронтенда. Если
   предложенный ИИ код не компилируется — не трать время на попытки
   понять, что не так, глядя глазами; читай сообщение компилятора, оно
   почти всегда указывает точную строку и колонку.
2. **Держи рядом «якорь» — реальный похожий файл в этом же
   репозитории.** Прежде чем поверить, что синтаксис, который дал ИИ,
   «типичный для проекта», открой соседний файл того же рода
   (например, другой `pgrepo.go` для сравнения с новым) — сверяй
   форму (именование, структуру импортов, стиль обработки ошибок) с
   ним, а не с абстрактным примером из общего интернета, который мог
   быть на другом стиле/версии языка.
3. **Проси объяснение «почему такой синтаксис», а не только код.**
   Если объяснение апеллирует к архитектурному свойству, которое ты
   уже понимаешь («это указатель, потому что нужно расшарить одно
   соединение с БД на все горутины, а не копировать его» — ты уже
   знаешь «shared mutable resource») — это хороший знак согласованности.
   Если объяснение сводится к «так принято» без связи со смыслом —
   переспроси или проверь по `../go-guide.md`.
4. **Запусти, не только прочитай.** У джуниора нет доверия к
   собственному чтению кода — поэтому он привычно всё запускает и
   смотрит на реальный результат. Заимствуй эту привычку сознательно:
   `curl` к живому эндпоинту, `console.log`/`fmt.Println`/`print()` в
   ключевых точках, реальный прогон теста — дешевле и надёжнее, чем
   вычитывание синтаксиса на веру.

## 5. Как встроить это в маршрут

В [`how-to-read-and-learn.md`](how-to-read-and-learn.md) день 1 и
неделя 1 расписаны как последовательность файлов. Твоя версия того же
маршрута:

- Раздел 1 «Что читать перед тем, как читать код» там — **пропускай
  почти всё**, кроме строки таблицы «Async/потоковые ответы (SSE)»,
  если раньше не работал с потоковыми HTTP-ответами буквально руками
  (не концептуально — ты и так знаешь, что такое стриминг данных).
- В каждом шаге дня 1/недели 1, когда написано «откройте файл X» —
  сначала посмотри на него в таблице раздела 1 этого документа (или
  найди похожую строку по смыслу), чтобы связать файл с уже знакомой
  тебе архитектурной ролью, и только потом читай сам код, держа под
  рукой раздел 2-3 как шпаргалку по конкретному синтаксису, который
  там встретится.
- Не пытайся выучить Go/TypeScript/Python «в отрыве», отдельным курсом
  до начала чтения кода — учи нотацию непосредственно на маршруте,
  используя этот документ как словарь по требованию, а не учебник по
  порядку.
- Раздел 4 (ИИ) здесь **заменяет**, а не дополняет раздел 4 в
  `how-to-read-and-learn.md` для твоего профиля — там расчёт на то, что
  человек не знает архитектуры и учится ей через ИИ; тебе нужен
  прямо противоположный вектор запросов.
