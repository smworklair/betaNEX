Скопируй всё, что ниже, как первое сообщение другой нейросети (или новой сессии). Он самодостаточен — не предполагает, что модель видела предыдущую переписку.

## Контекст проекта

Проект — **betaNEX** («КИС Колледж»): Go-бекенд `nexd` (модульный монолит) + React/Vite фронтенд. Пользователь — единственный админ, общается по-русски, крайне требователен к тому, чтобы UI/UX выглядел цельно и не «сыро», и **не терпит, когда его просят подтвердить очевидное вместо того, чтобы сразу делать и проверять результат в браузере/тестами**.

Ветка для работы: `claude/backend-session-improvements-mfonkr`. Коммитить и пушить в неё, PR не создавать без явной просьбы.

**Важно:** ты запускаешься **локально на ПК пользователя**, не в облачной песочнице — репозитория и окружения ещё нет, их нужно поднять с нуля.

### Локальная настройка окружения (сделать первым делом)

1. Клонировать репозиторий и переключиться на рабочую ветку:
   ```sh
   git clone <URL этого репозитория betaNEX> betaNEX && cd betaNEX
   git fetch origin claude/backend-session-improvements-mfonkr
   git checkout claude/backend-session-improvements-mfonkr
   ```
   Если ветки на remote ещё нет — создать её от `main`/`master` с этим именем и запушить после первого коммита.
2. **Бекенд (Go 1.25+)**:
   - `go build ./...` и `go test ./...` должны проходить сразу после клонирования (baseline перед изменениями).
   - Полноценный запуск с БД: `make dev` (поднимает Postgres 16+ через `docker compose`, нужен установленный Docker), затем `make run` (или `go run ./cmd/nexd`). Без Docker — можно и без базы: не задавай `NEX_DATABASE_URL`, `nexd` подхватит **in-memory режим** (без сохранения данных, но модули и HTTP работают, этого достаточно для разработки и большинства ручных проверок).
   - `make help` — список всех make-целей (`make build`, `make test`, `make test-db` и т.д.), если Makefile непонятен — открой его, не гадай.
3. **Фронтенд (Node.js + pnpm)**:
   - `cd web && pnpm install`.
   - `pnpm dev` — dev-сервер Vite на `http://localhost:3000` (или порт из вывода команды). Без `web/.env` фронт работает в **демо-режиме** (моки, сеть не трогается) — этого достаточно для проверки UI/UX терминала.
   - Чтобы фронт реально ходил на локально запущенный `nexd`: скопировать `web/.env.example` → `web/.env`, раскомментировать и указать `VITE_API_URL=http://localhost:8080` (порт nexd по умолчанию — проверить в конфиге/логах при запуске).
   - `pnpm lint` (это `tsc --noEmit`) и `pnpm build` — обязательные проверки перед коммитом.
4. Проверять реальный результат в браузере на `localhost` (можно обычным браузером на этом же ПК — headless/Playwright не обязателен, если недоступен локально, но крайне желателен хоть какой-то визуальный прогон перед тем, как считать задачу готовой).

### Архитектура бекенда (не ломать)

- `internal/kernel/` — ядро: identity/authz (RBAC), command bus (`command.Bus`), audit (append-only журнал, диффы полей), tenancy.
- `internal/module/<name>/` — модули (tasks, finance, campus, files, notifications, terminal…). Модуль **не импортирует соседние модули** — всё нужное приходит адаптерами (функциями) из composition root.
- `cmd/nexd/main.go` — composition root: единственное место, где всё собирается вместе.
- Правило железное: **чтения** — через репозитории модулей напрямую (за guard'ом прав), **мутации** — только через шину команд (`bus.Dispatch`), которая сама авторизует по праву команды и пишет аудит в одной транзакции с изменением данных. Никаких прямых SQL/сервисных вызовов в обход шины для мутаций.
- Деньги — `int64` в копейках. Права — строки вида `module:action` (`terminal:exec`, `tasks:write`…), выдаются ролям в `main.go`.

### Терминал сегодня — что уже есть

Бекенд: `internal/module/terminal/`
- `terminal.go` — тип `Deps` (поля-функции-адаптеры на кампус/финансы/задачи/аудит/пользователей), `Result` (структурированный ответ: `kind: text|table|kpi`, `Title`, `Text`, `Columns`, `Rows`, `KPIs`, `Hint`), карта `canon` (алиасы команд, включая русские: `обзор`, `аналитика`, `финансы`, `безопасность`, `задачи`, `новая задача <текст>`, `готово <id>`, `уведомить`, `аудит [n]`, `кто я`…), метод `Exec(ctx, line) (Result, error)`.
- `http.go` — `POST /api/v1/terminal/exec`, право `terminal:exec` (только роль admin).
- Домены реализованы на настоящих данных: Аналитика (campus — группы/студенты/оценки), Финансы (ledger — счета/проводки), Безопасность (пользователи+аудит), Задачи, обзор/whoami.
- Тесты есть (`terminal_test.go`), `go build`/`go test` зелёные.

Фронтенд:
- `web/src/terminal.tsx` — единый реестр `TERM_COMMANDS: TermCommand[]` (демо-режим, работает поверх тех же локальных данных, что и обычные страницы сайта), функции `execRegistry(line, ctx)` и `suggest(input)` (префиксные подсказки), компонент `TerminalWorkspace` — **полноэкранная среда терминала** (портал в `document.body`, открывается кнопкой ⛶ из компактного блока, Esc закрывает), рендер `TermResBlock`.
- `web/src/pages/Home.tsx` — компактный блок терминала на «Главном» (`NexTerminal`), использует тот же `execRegistry`; при заданном `VITE_API_URL` часть команд (`TERMINAL_BACKEND_TOKENS` из `web/src/api/terminal.ts`) уходит на настоящий бекенд через `terminalExec()`, при ошибке — честный фолбэк на демо с пометкой.
- `web/src/api/terminal.ts` — клиент `terminalExec()` + `TERMINAL_BACKEND_TOKENS` (список первых слов команд, которые понимает бекенд — держать в синхроне с картой `canon` в Go).
- Домены: `Обзор`, `Аналитика`, `Финансы`, `Безопасность`, `Задачи` (`TermDomain` в `terminal.tsx`, рейл слева в `TerminalWorkspace`).
- UX уже сделан: метка `время · длительность · сервер/демо` под каждым ответом, живой «пульс системы» в рейле, шпаргалка горячих клавиш (Tab/↑↓/Enter/Esc), история команд по ↑/↓, входные анимации блоков, ховеры таблиц.

### Важные соглашения UX/дизайна (не ломать простыми правками)

1. **Все ИИ-поверхности (мини-чаты, консоль) — приглушённые по цвету.** Акцентный синий (`var(--ai)`/`var(--accent)`) — только на маленьких деталях (иконка-«орб», акцентный текст команды), никогда не заливка всего блока. Есть готовый класс `.ai-orb` (контейнерный бейдж иконки) — переиспользуй, не изобретай новый треугольник дизайна.
2. **Мини-чаты** (`nex-inline`, fin-copilot, AiBox) специально сделаны похожими на обычный чат (пузыри вопрос/ответ), но в приглушённых, а не кричащих цветах, и **раскрываются в потоке своей карточки**, а не поверх контента и не раздувают соседние карточки в CSS-гриде (см. класс `.inline-row { align-items: start; }` на контейнере ряда).
3. Терминал — **отдельная, самостоятельная поверхность**, не «ИИ-чат с элементами терминала». Ответы — структурированные данные (таблица/KPI/текст), а не проза; никакой имитации печатающегося ответа как у чат-бота.
4. У готовой командной палитры сайта (⌘K, компонент `omni-pop`/`cmd-item` в `App.tsx`/`index.css`) уже есть отработанный визуальный язык списка команд с подсказками и секциями — **при добавлении палитры команд терминала (задача 1 ниже) в первую очередь проверь, можно ли переиспользовать этот же визуальный язык** (те же классы или явный вариант на их основе), а не выдумывать третий стиль списка в приложении.
5. Everything по-русски, без сложного синтаксиса флагов (`--like-this`) — админ не программист, команды это человеческие слова с опциональным аргументом.

## Задача 1 (короткая, сделать первой): командная палитра по «/»

В строке ввода терминала (и в компактном блоке на «Главном», и в `TerminalWorkspace`) по вводу символа `/` (в начале пустой строки, аналогично тому, как `⌘K` открывает омнибокс) должен всплывать список **всех** доступных команд терминала — не только тех, что начинаются на введённый текст (как сейчас делает `suggest()`), а полный реестр, сгруппированный по доменам (`Обзор`/`Аналитика`/`Финансы`/`Безопасность`/`Задачи`), с кратким описанием каждой и, если есть — форматом аргумента. Клик/Enter по пункту — либо сразу выполняет команду (если без аргумента), либо подставляет её в строку ввода с пробелом в конце (если аргумент нужен) и оставляет фокус в поле, чтобы админ дописал аргумент.

Технически: `TERM_COMMANDS` уже содержит всё нужное (`domain`, `id`, `aliases`, `arg`, `desc`) — просто нужен режим отображения «показать всё, сгруппированное по `domain`», а не текущий `suggest()` (префиксный, плоский, максимум 6 штук). Не удаляй `suggest()` — он остаётся для обычного набора текста; `/` — отдельный, дополняющий режим.

## Задача 2 (крупная, главная): терминал = операционная система управления организацией

Пользователь просит **сам додумать и реализовать** максимально проработанные бекенд и UX — не спрашивать «а что именно добавить», а предложить и сделать content-complete итерацию, ориентируясь на то, что уже построено. Общее направление, которое админ подтвердил в предыдущих раундах: терминал — не чат, а **AI-native способ управлять всем функционалом сайта без обычной навигации**, при этом обычный сайт (карточки/страницы) продолжает существовать параллельно, ничего не удаляется.

Ожидаемый масштаб проработки (используй как чек-лист, расширяй по вкусу, если видишь явно недостающий домен управления организацией — добавляй, не спрашивая):

- **Больше доменов и команд** внутри уже существующих (`Аналитика`/`Финансы`/`Безопасность`/`Задачи`) и, где оправдано, новые домены (например «Люди»/«Кадры», «Расписание», «Кампус/Учёба», «Уведомления»), если для них уже есть модули в бекенде (`internal/module/*`) — не выдумывай данные, бери из реальных репозиториев модулей так же, как уже сделано для campus/finance.
- **Мутирующие команды** (не только чтение): создание/изменение сущностей там, где это безопасно и укладывается в один короткий человеческий аргумент (по образцу `новая задача <текст>`, `готово <id>`, `уведомить <получатель> <текст>`) — обязательно через шину команд, с валидацией, правом и аудитом, как всё остальное в системе.
- **Разрешение неоднозначности** — если аргумент команды похож на несколько сущностей (например email похож на двух пользователей, или название группы неточное), команда должна вернуть уточняющий список, а не падать с ошибкой или молча брать первое совпадение.
- **Композиция команд / контекст**: рассмотри, нужна ли последовательная работа с «выбранной» сущностью в рамках сессии терминала (например: `студенты иванов` → показал одного → следующая команда неявно про него), не усложняя интерфейс — если это добавит путаницы, лучше не делать.
- **UX-глубина**, аналогично уже сделанному: тайминги, статус источника данных, история, подсказки — расширяй тем же языком, не изобретай параллельный набор паттернов. Обрати особое внимание на:
  - что видно **новому** пользователю терминала, который ни разу не работал с консолью (онбординг первого экрана, не только `/`-палитра);
  - как обозначается разница между «безопасной» командой (чтение) и «действующей» (мутацией) — например, лёгкий визуальный маркер или подтверждение для деструктивных действий, если такие появятся;
  - отзывчивость на маленьких экранах (сейчас `TerminalWorkspace` уже адаптируется на `max-width: 720px`, но при разрастании списка команд это нужно перепроверить).

### Обязательные проверки перед тем, как считать итерацию готовой

1. `go build ./...` и `go test ./...` — зелёные.
2. `pnpm -C web lint` (tsc) и `pnpm -C web build` — зелёные.
3. Реальный прогон в браузере на `localhost` (запусти `pnpm -C web dev`, открой руками или через Playwright/иной headless-браузер, если он у тебя есть локально) новых команд и палитры — не полагайся только на типы/сборку. В прошлых итерациях именно так ловились реальные баги вёрстки (например, оверлей вместо расширения блока). Если совсем нет способа отрендерить браузер в своём окружении — явно скажи об этом пользователю в конце вместо того, чтобы молча пропустить проверку.
4. Не создавай PR без явной просьбы; коммить и пуш в `claude/backend-session-improvements-mfonkr` — это ожидаемый рабочий цикл (сначала `git pull --rebase` перед пушем, если в это время кто-то ещё коммитил).

Если чего-то не хватает в контексте (например, не находишь модуль под нужный домен, или неясно, что считать «безопасной» мутацией) — прими наиболее консервативное разумное решение сам и объясни его одной строкой в коммите; не блокируйся на уточняющих вопросах, если ответ можно вывести из уже принятых в проекте решений (см. раздел «соглашения» выше).
