# NEX — Журнал технических решений

Формат: выбор → почему → что отклонено → плюсы → минусы. Решения пересматриваются, если меняются вводные.

## Backend

### ADR-001. Язык: Go

- **Почему:** уже принят в проекте; идеален для модульного монолита — один статический бинарник, встроенная конкурентность, быстрая компиляция, стабильность API языка (совместимость Go 1.x).
- **Отклонено:** Node.js/TypeScript (слабее для долгоживущего ядра: runtime-ошибки, тяжёлый деплой), Python (медленный, GIL), Rust (дорогая разработка, избыточен для CRUD-ядра), Java/Kotlin (тяжёлая инфраструктура для маленькой команды).
- **Плюсы:** производительность, простой деплой, лучшая в индустрии обратная совместимость, огромный пул библиотек для web.
- **Минусы:** многословность, дженерики ограничены, меньше «магии» — больше ручного кода.

### ADR-002. Без web-фреймворка: `net/http` + `ServeMux`

- **Почему:** код уже написан в этом стиле (и написан хорошо); с Go 1.22 роутер stdlib умеет методы и path-параметры — 90% ценности chi бесплатно; ноль зависимостей в самом критичном слое.
- **Отклонено:** chi (хороший, но сейчас не даёт достаточно поверх stdlib; допустимо добавить позже ради вложенных роутеров), Gin/Echo (собственные типы контекста → привязка к фреймворку), Fiber (несовместим с net/http, fasthttp — риск).
- **Плюсы:** переживёт любой фреймворк, любой Go-разработчик поймёт код, middleware — обычные `http.Handler`.
- **Минусы:** нет групп роутов и удобств из коробки; часть утилит пишется руками.

### ADR-003. API: REST + OpenAPI 3.1 (`oapi-codegen`), ошибки RFC 9457

- **Почему:** контракт-first устраняет рассинхрон фронта и бэка: одна спека генерирует и Go-сервер (интерфейсы + валидация), и TS-клиент. REST достаточно для CRUD-доменов КИС.
- **Отклонено:** gRPC (браузер требует прокси; выгоден для межсервисного общения, которого в монолите нет), GraphQL (сложность резолверов/N+1/кэша не окупается при одном клиенте), tRPC (привязал бы бэкенд к TS).
- **Плюсы:** типобезопасность на обоих концах, автодокументация, простое тестирование curl'ом.
- **Минусы:** дисциплина «сначала спека», кодогенерация — лишний шаг сборки.

### ADR-004. Аутентификация: собственная, opaque-сессии + argon2id

- **Почему:** identity — часть kernel по архитектуре NEX; server-side сессии дают мгновенный отзыв (критично для учебных заведений), argon2id — текущая рекомендация OWASP.
- **Отклонено:** JWT как сессии (нельзя отозвать, соблазн класть данные в токен), Keycloak/Ory (тяжёлая внешняя зависимость против идеи самодостаточного монолита), Auth0/Clerk (SaaS, деньги, данные студентов у третьей стороны).
- **Плюсы:** полный контроль, отзыв сессий, простая модель, нет внешних сервисов.
- **Минусы:** пишем и сопровождаем сами; безопасность — наша ответственность (митигируется OWASP ASVS-чеклистом в роадмапе). OIDC-федерацию добавим позже отдельным слоем.

### ADR-005. Авторизация: собственный RBAC в kernel

- **Почему:** архитектура NEX явно помещает authorization в kernel; RBAC с tenant-скоупами покрывает домен (роли: администратор, преподаватель, студент…); проверки в command-слое дают аудит «кто, что, почему разрешено».
- **Отклонено:** Casbin (внешний DSL-язык политик сложнее самих политик), OPA (отдельный сервис и Rego — избыточно), только Postgres RLS (RLS оставляем как второй рубеж, но не как единственный механизм — политики в SQL трудно тестировать и объяснять).
- **Плюсы:** политики — обычный Go-код, тестируемый юнит-тестами; единая точка enforcement.
- **Минусы:** рост сложности при переходе к ABAC/ReBAC — тогда пересмотрим (кандидат: openFGA).

### ADR-006. Доступ к БД: `sqlc` + `pgx/v5`, без ORM

- **Почему:** sqlc генерирует типобезопасный Go из настоящего SQL — вся мощь Postgres (CTE, RLS, оконные функции) без слоя абстракции; pgx — фактический стандарт драйвера.
- **Отклонено:** GORM (скрывает SQL, сюрпризы производительности, рефлексия), Ent (кодогенерация схемы мощная, но своя графовая модель — лишняя концепция), goqu/squirrel (строители запросов — теряется проверяемость SQL).
- **Плюсы:** SQL на виду и проверяется на этапе генерации, производительность драйвера, тривиальный переход на любые фичи Postgres.
- **Минусы:** динамические запросы (сложные фильтры) требуют ручного кода; миграции отдельно (goose).

### ADR-007. БД: PostgreSQL 17, tenant_id + RLS

- **Почему:** одна СУБД покрывает всё: реляционка, JSONB, полнотекст, LISTEN/NOTIFY, очередь (River). Multi-tenancy строкой `tenant_id` + RLS — проще пула схем, бэкапится одним махом.
- **Отклонено:** MySQL (слабее по фичам: RLS, JSONB, transactional DDL), MongoDB (домен реляционный: студенты-группы-оценки), SQLite (не для multi-tenant сервера), schema-per-tenant (операционный ад миграций при десятках колледжей).
- **Плюсы:** зрелость, документация, один компонент вместо трёх (БД+очередь+частично кэш).
- **Минусы:** RLS требует дисциплины (`SET app.tenant_id` в каждой транзакции); вертикальное масштабирование до определённого предела — для КИС это не риск.

### ADR-008. Кэш: сначала in-process, Valkey — по необходимости

- **Почему:** монолит в один инстанс не нуждается в сетевом кэше; преждевременный Redis — лишний компонент и лишний класс багов (инвалидация).
- **Отклонено (пока):** Redis (лицензия RSAL; если понадобится сетевой кэш — берём Valkey, open-source форк), Memcached (беднее функционально).
- **Плюсы:** нулевая инфраструктура на старте; понятный триггер апгрейда (второй инстанс/тяжёлые запросы).
- **Минусы:** при горизонтальном масштабировании потребуется миграция на Valkey — интерфейс кэша в коде закладываем сразу.

### ADR-009. Фоновые задачи: River

- **Почему:** очередь поверх Postgres с транзакционным enqueue — задача ставится в той же транзакции, что и бизнес-операция: не теряется и не выполняется до коммита. Идеально ложится на Commands→Events спайн. Нет нового инфраструктурного компонента.
- **Отклонено:** Asynq (требует Redis), gocraft/work (заброшен), самописный воркер на LISTEN/NOTIFY (изобретение велосипеда: ретраи, дедлайны, приоритеты).
- **Плюсы:** надёжность транзакций, UI для наблюдения, активная разработка, pgx-нативность.
- **Минусы:** нагрузка на Postgres (для объёмов КИС несущественно); относительно молодой проект — но с сильными мейнтейнерами (brandur, ex-Stripe/Heroku).

### ADR-010. Логи/метрики/трейсы: slog + Prometheus + OpenTelemetry

- **Почему:** slog уже внедрён; Prometheus — стандарт метрик самостоятельного хостинга; OTel даёт trace-id через весь запрос (HTTP → command → SQL → job).
- **Отклонено:** zap/zerolog (быстрее slog, но stdlib достаточно и без зависимостей), коммерческие APM (Datadog — деньги и данные наружу).
- **Плюсы:** стандарты, самостоятельный хостинг, нулевой вендор-лок.
- **Минусы:** OTel-инструментирование добавляет шума в код — вводим на этапе наблюдаемости, не раньше.

## Frontend

### ADR-011. React 19 + Vite + TypeScript (сохраняем)

- **Почему:** уже в проекте; крупнейшая экосистема; текущий прототип переиспользуем как референс дизайна.
- **Отклонено:** Vue/Svelte (переучивание без выгоды), Next.js (SSR не нужен внутренней системе за логином; SPA проще деплоить рядом с Go-бинарником).
- **Плюсы:** экосистема, кадры, совместимость с shadcn/TanStack.
- **Минусы:** React — не самый быстрый рантайм; для админ-интерфейса неважно.

### ADR-012. Состояние: TanStack Query + Zustand

- **Почему:** 90% состояния КИС — серверные данные; Query решает кэш/инвалидацию/ретраи/optimistic updates. Для остального UI-состояния (сайдбар, фильтры) хватает Zustand без бойлерплейта.
- **Отклонено:** Redux Toolkit (церемония ради церемонии при наличии Query), MobX (магия реактивности), только Context (ре-рендеры, нет кэша).
- **Плюсы:** резкое сокращение кода загрузки данных; devtools; проверенная связка.
- **Минусы:** две библиотеки состояния — требует дисциплины «серверное — в Query, клиентское — в Zustand».

### ADR-013. Роутинг: TanStack Router

- **Почему:** единственный роутер с полной типизацией params/search/loaders end-to-end; search-params как первоклассное состояние — идеально для фильтров в таблицах КИС.
- **Отклонено:** React Router v7 (слабее типизация; режим framework тянет в SSR), собственный роутер прототипа (не масштабируется).
- **Плюсы:** типобезопасность, интеграция с Query (loader → prefetch), file-based routes по желанию.
- **Минусы:** моложе React Router; больше концепций на входе.

### ADR-014. UI: Tailwind v4 + shadcn/ui, иконки lucide, анимации Motion

- **Почему:** Tailwind и lucide уже в проекте; shadcn/ui — не зависимость, а копируемый код на Radix-примитивах: полный контроль для собственной дизайн-системы NEX; Motion уже используется.
- **Отклонено:** MUI/Ant Design (чужой дизайн-язык, тяжёлая тема), Chakra (рантайм-CSS против utility-first), CSS Modules (медленнее итерации).
- **Плюсы:** доступность из Radix бесплатно, дизайн-токены свои, ничего не «перекрашиваем».
- **Минусы:** shadcn-код надо сопровождать самим (это и фича); Tailwind-классы в JSX многословны.

### ADR-015. Формы и валидация: React Hook Form + Zod

- **Почему:** RHF — минимум ре-рендеров и стандарт де-факто; Zod-схемы переиспользуются в формах и при парсинге ответов API; отличная интеграция с TanStack Router (валидация search-params).
- **Отклонено:** Formik (стагнирует), Yup (слабее вывод типов), Valibot (легче, но экосистема меньше — допустимая альтернатива).
- **Плюсы:** одна схема — формы + API + роутер; производительность.
- **Минусы:** Zod-схемы дублируют часть OpenAPI-контракта — минимизируется генерацией типов из спеки.

## DevOps

### ADR-016. pnpm вместо npm

- **Почему:** быстрее, строгая изоляция зависимостей (ловит phantom dependencies), экономия диска.
- **Отклонено:** npm (медленный, плоский node_modules), yarn (нет преимуществ над pnpm), bun (рантайм-амбиции не нужны, зрелость ниже).
- **Плюсы/минусы:** стандарт индустрии 2026 / требует `corepack enable` у каждого разработчика.

### ADR-017. Линт/формат: golangci-lint + gofumpt; Biome для TS

- **Почему:** golangci-lint — агрегатор всех Go-линтеров; Biome заменяет ESLint+Prettier одним быстрым инструментом, конфиг в 10 строк.
- **Отклонено:** ESLint+Prettier (два инструмента, медленно, конфиг-ад; выбрать можно, если понадобятся специфичные плагины), oxlint (быстро растёт, но правил пока меньше — кандидат на пересмотр).
- **Минусы Biome:** нет части узких ESLint-плагинов (a11y покрыт частично).

### ADR-018. Деплой: Docker Compose на VPS + Caddy

- **Почему:** масштаб КИС-колледжа — это один-два сервера; Compose воспроизводим и понятен; Caddy даёт авто-TLS и раздачу статики фронта в 5 строк конфига.
- **Отклонено:** Kubernetes (операционная стоимость несопоставима с масштабом), PaaS Fly/Render (данные студентов лучше держать на контролируемом сервере; расходы), bare-metal без контейнеров (невоспроизводимо).
- **Плюсы:** дёшево, просто, миграция на k8s возможна позже (образы уже есть).
- **Минусы:** ручное управление сервером — компенсируем runbook'ом и мониторингом.

### ADR-019. CI/CD: GitHub Actions

- **Почему:** репозиторий на GitHub; бесплатно для приватных реп в разумных лимитах; огромная база actions.
- **Отклонено:** GitLab CI (переезд репы), Jenkins (сопровождение сервера CI), Drone/Woodpecker (self-hosted — позже, если лимиты).
- **Минусы:** вендор-лок на GitHub умеренный — пайплайн простой, переносится за день.

### ADR-020. Мониторинг: Prometheus + Grafana + Loki, GlitchTip, Uptime Kuma

- **Почему:** полностью self-hosted стек, стандарт индустрии; GlitchTip совместим с Sentry SDK (и Go, и браузер).
- **Отклонено:** Sentry SaaS/Datadog (деньги, данные наружу), ELK (тяжёлый — Loki проще на порядок).
- **Минусы:** свой стек надо поднять и сопровождать — входит в веху «Мониторинг» роадмапа.

### ADR-021. Терминал «Администратор · альфа»: консоль как AI-native вход во всю систему

- **Почему:** администратору нужна одна точка управления всем функционалом без прыжков по разделам; терминал уже жил на «Главном» как чат — превращаем его в консоль экосистемы: `status/tasks/task add|done/users/notify/audit/whoami` + свободные вопросы к ИИ. Бэкенд-модуль `internal/module/terminal` не импортирует соседние модули — все зависимости приходят адаптерами из composition root; чтения идут через модульные репозитории, мутации — только через шину команд (авторизация + аудит + транзакция), то есть терминал — ещё один actor на общем командном позвоночнике, как и ИИ.
- **Отклонено:** прямые SQL/сервисные вызовы из обработчика терминала (обход авторизации и аудита), выделенный gRPC-протокол (оверкилл для альфы), исполнение свободного текста LLM-ом на сервере (команды детерминированы, LLM остаётся на фронте для вопросов).
- **Плюсы:** одна поверхность для всего функционала, структурированные ответы (text/table/kpi) вместо прозы, право `terminal:exec` только у admin.
- **Минусы:** парсер команд примитивен (strings.Fields) — расширим по мере роста; демо-режим фронта дублирует команды локальным движком (цена работы без бэкенда).
