# NEX — Роадмап: от альфы к продакшену

Текущее состояние: работающий каркас `nexd` (config, HTTP-сервер, healthz, slog, graceful shutdown, тесты) + фронтенд-прототип «КИС Колледж» (референс дизайна, не архитектура).

Правила: вехи идут по порядку, но M7 (frontend) может идти параллельно с M4–M6. Веха закрыта, когда её результат покрыт тестами и прошёл CI.

## M0. Порядок в репозитории

- **Цель:** структура, в которой не стыдно жить два года.
- **Задачи:** перенести прототип из `src/` + `index.html` в `/web/` (или в отдельную ветку `prototype`); удалить артефакты AI Studio (`metadata.json`, Gemini-переменные из `.env.example`); добавить `.editorconfig`, `LICENSE`, `CONTRIBUTING.md`; настроить `golangci-lint` + конфиг; CI-скелет (lint + test на PR); перейти на pnpm.
- **Зависимости:** нет.
- **Результат:** чистый корень = Go-сервис; каждый PR автоматически проверяется.

## M1. Платформа данных

- **Цель:** сервис умеет хранить данные.
- **Задачи:** `compose.yaml` (Postgres 17, позже Valkey); `internal/platform/postgres` — пул pgx, health-пинг; миграции goose (embed) + `make migrate`; каркас sqlc; endpoint `/readyz` (готовность = БД доступна); testcontainers-go для интеграционных тестов.
- **Зависимости:** M0.
- **Результат:** `make dev` поднимает окружение; тесты гоняются против реального Postgres.

## M2. Kernel: tenancy + Commands→Events→Audit

- **Цель:** несущая конструкция архитектуры NEX.
- **Задачи:** таблицы tenants; протаскивание tenant-контекста через middleware → команды → SQL (`SET app.tenant_id`), RLS-политики; шина команд (валидация, авторизация-хук, транзакция); доменные события + append-only журнал аудита; интеграция River (события → фоновые задачи).
- **Зависимости:** M1.
- **Результат:** любое изменение данных проходит один аудируемый путь; изоляция tenant'ов проверена тестами (негативные сценарии обязательны).

## M3. Аутентификация

- **Цель:** kernel знает, кто пришёл.
- **Задачи:** users + credentials (argon2id); сессии: opaque-токен, httpOnly secure cookie, ротация, отзыв; endpoints login/logout/refresh/me; rate limiting и защита от перебора; аудит входов.
- **Зависимости:** M2.
- **Результат:** защищённые маршруты работают; чеклист OWASP Session Management пройден.

## M4. Авторизация

- **Цель:** kernel знает, что пришедшему можно.
- **Задачи:** модель ролей и прав (per-tenant); enforcement в шине команд (не в HTTP-хендлерах); предзаданные роли домена (админ, преподаватель, студент); отражение решений в аудите; тестовая матрица «роль × команда».
- **Зависимости:** M3.
- **Результат:** ни одна команда не выполняется без явной проверки прав.

## M5. Контракт API v1

- **Цель:** стабильный, типизированный публичный интерфейс.
- **Задачи:** `api/openapi.yaml` как источник истины; oapi-codegen для сервера; формат ошибок RFC 9457; пагинация/фильтры/сортировка — единые конвенции; версионирование `/api/v1`; генерация TS-клиента (`openapi-typescript`); Bruno-коллекция в репозитории.
- **Зависимости:** M2 (частично параллельно M3–M4).
- **Результат:** фронтенд разрабатывается против контракта, а не против кода.

## M6. Первый модуль домена

- **Цель:** доказать модульную архитектуру реальным доменом колледжа.
- **Задачи:** модуль `college`: студенты, группы, преподаватели, дисциплины (минимальный срез); полный цикл: миграции → sqlc → команды/события → API; фоновая задача через River (например, импорт списков); документация «как написать модуль».
- **Зависимости:** M4, M5.
- **Результат:** рабочий вертикальный срез от HTTP до БД; шаблон для всех будущих модулей.

## M7. Frontend: новая основа + дизайн-система

- **Цель:** заменить прототип на сопровождаемое приложение.
- **Задачи:** новый каркас в `/web`: Vite + TS strict + TanStack Router/Query + Zustand + Biome; дизайн-токены (цвета/типографика из прототипа) поверх Tailwind v4 + shadcn/ui; базовые компоненты: layout, таблица (TanStack Table), формы (RHF+Zod); интеграция auth (login, guard-роуты, интерцептор 401); экраны модуля college поверх сгенерированного API-клиента.
- **Зависимости:** M5 (контракт); дизайн-часть — параллельно M4–M6.
- **Результат:** прототип полностью выведен из эксплуатации.

## M8. Тестирование как система

- **Цель:** уверенность в изменениях без ручного прогона.
- **Задачи:** цели покрытия (kernel ≥ 80%); интеграционные тесты всех команд (testcontainers); e2e-смоук Playwright (логин → CRUD студента); k6-сценарий базовой нагрузки; `govulncheck` и аудит зависимостей в CI; обязательный зелёный CI для merge.
- **Зависимости:** M6, M7.
- **Результат:** релиз = зелёный пайплайн, а не «вроде работает».

## M9. Поставка и деплой

- **Цель:** воспроизводимый путь до сервера.
- **Задачи:** Dockerfile (multi-stage, distroless) для nexd; сборка фронта → раздача Caddy; прод-`compose.yaml` (nexd, caddy, postgres); staging-VPS; деплой по git-тегу из GitHub Actions; бэкапы: pg_dump ежедневно + pgBackRest, офсайт; процедура restore, проверенная руками.
- **Зависимости:** M8.
- **Результат:** `git tag v0.x` → staging обновлён; бэкап восстановлен хотя бы один раз.

## M10. Наблюдаемость

- **Цель:** видеть систему до того, как позвонит пользователь.
- **Задачи:** Prometheus-метрики (`/metrics`: RPS, латентность, ошибки, пул БД, очередь River); Grafana-дашборд + алерты (телеграм/почта); Loki для логов; GlitchTip для ошибок (Go + браузер); OTel-трейсы HTTP → command → SQL; Uptime Kuma.
- **Зависимости:** M9.
- **Результат:** алерт приходит раньше жалобы; у каждого инцидента есть trace.

## M11. Hardening и безопасность

- **Цель:** система готова хранить данные реальных студентов.
- **Задачи:** прогон OWASP ASVS L1→L2; заголовки безопасности, CSRF-стратегия, CORS; лимиты и таймауты повсюду; ревизия RLS-политик; runbook инцидентов; политика обновления зависимостей (dependabot); соответствие требованиям к персональным данным (152-ФЗ, если применимо — хранение в РФ).
- **Зависимости:** M10.
- **Результат:** заполненный ASVS-чеклист; известные риски задокументированы.

## M12. Продакшен-релиз

- **Цель:** первый колледж работает в системе.
- **Задачи:** прод-VPS по образу staging; миграция/загрузка реальных данных; пилот с ограниченной группой пользователей; SLO (аптайм, латентность) и еженедельный обзор метрик; план отката; обучение администратора колледжа.
- **Зависимости:** M11.
- **Результат:** v1.0 в эксплуатации; цикл «фича → тесты → тег → деплой» занимает меньше дня.

## После v1.0 (кандидаты)

Второй модуль домена (расписание, оценки); OIDC-федерация; полнотекстовый поиск (Postgres FTS); Valkey при втором инстансе; экспорт отчётов (xlsx/pdf); AI-актор через тот же command-путь — как и задумано архитектурой.
